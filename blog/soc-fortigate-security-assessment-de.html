55.279 Geloggte Events
3 Kritische Lücken

Einleitung

In der vorherigen Traffic-Analyse haben wir die Fortigate-Logs auf C2-Kommunikation, SMB-Traffic und externe Verbindungen untersucht. Jetzt stellen wir die entscheidende Frage: Hat die Firewall ihre Aufgabe erfüllt?

🚨

Kurz gesagt: Nein. Die Fortigate hat nur 20% der erwarteten Sicherheitsfunktionen erfüllt. C2-DNS-Traffic passierte ungehindert, SMB-Lateral-Movement war vollständig unsichtbar, und die einzigen blockierten Pakete waren Broadcasts — etwas, das jede Firewall standardmäßig macht.

GAP 1: C2-DNS-Bypass 🔴

Das Problem

Die Cerber-Ransomware auf 192.168.250.20 führte 1.194 DNS-Lookups zu 8.8.8.8 (Google Public DNS) durch. Die Firewall-Action: accept.

Warum ist das ein Problem?

  • DNS ist Port 53 — in den meisten Netzwerken erlaubt
  • Google DNS (8.8.8.8) ist "vertrauenswürdig" und wird selten blockiert
  • Die Ransomware nutzt DNS, um C2-Domains aufzulösen (z.B. .onion-Exit-Nodes)
  • Fortigate DNS Filter war nicht aktiv oder nicht konfiguriert
❌

Fortigate Versagen: Eine moderne Firewall muss DNS-Filtering beherrschen — nicht nur Port-basiertes Blocking. Ohne DNS-Filter ist jede Malware mit C2-Domain-Kommunikation unsichtbar, selbst wenn sie über "vertrauenswürdige" Resolver wie 8.8.8.8 geht.

Was hätte passieren sollen?

FORTIGATE CLI DNS Filter Konfiguration für C2-Blocking
# DNS Filter Profil erstellen config system dns-filter-profile edit "C2_DNS_Block" config ftgd-dns-filters edit 1 set category 26 # Malicious Websites set action block next edit 2 set category 61 # Botnet set action block next end set log-all-domain enable next end # DNS Filter auf Firewall-Policy anwenden config firewall policy edit 1 set dnsfilter-profile "C2_DNS_Block" next end

GAP 2: SMB Blind Spot 🔴

Das Problem

Die SMB-Analyse zeigte 151.568 SMB-Events und Lateral-Movement (192.168.250.100 → 192.168.250.20). In den Fortigate-Logs erscheint kein SMB-Traffic.

Warum ist SMB unsichtbar?

  • Fortigate ist als Perimeter-Firewall positioniert (WAN → LAN)
  • SMB-Lateral-Movement passiert intern (East-West) im 192.168.250.x Subnet
  • Ohne Internal Segmentation sieht die Firewall kein internes SMB
  • Fortigate IPS hat keine SMB-Signatur (MS17-010/EternalBlue) ausgelöst
❌

Fortigate Versagen: Eine Firewall, die nur am Perimeter steht, ist für Lateral-Movement-Detection nutzlos. SMB-Exploits (EternalBlue, MS17-010) müssen an jedem Segment-Gateway erkannt werden — nicht nur am WAN.

Was hätte passieren sollen?

FORTIGATE CLI Internal Segmentation + SMB IPS
# 1. VLAN-Interface für interne Segmente config system interface edit "VLAN_250" set vdom "root" set ip "192.168.250.1 255.255.255.0" set interface "port2" set vlanid 250 next end # 2. IPS-Sensor für SMB-Exploits config ips sensor edit "SMB_Lateral_Movement" config entries edit 1 set action block set log enable set severity high set application "SMB" set location internal next end next end # 3. Inter-VLAN Policy mit IPS config firewall policy edit 100 set name "Internal_Segmentation" set srcintf "VLAN_250" set dstintf "VLAN_100" set ips-sensor "SMB_Lateral_Movement" set logtraffic all next end

GAP 3: Nur Broadcast-Denies 🟡

Was funktionierte?

Die einzigen deny-Events waren Broadcast-Traffic:

srcip dstip action count Bewertung
192.168.69.104 192.168.69.255 deny 1.498 🟢 Standard-Verhalten
192.168.69.104 224.0.0.1 deny 1.472 🟢 Standard-Verhalten
192.168.69.103 255.255.255.255 deny 356 🟢 Standard-Verhalten

Broadcast-Denies sind das Minimum. Jede Firewall blockiert Broadcasts über Subnet-Grenzen hinweg — das ist kein Erfolg, sondern Baseline-Verhalten.

⚠️

Falsche Sicherheit: Ein SOC-Team, das nur auf Firewall-Denies schaut, würde denken: "Alles gut, die Firewall blockiert ja." Tatsächlich wurde der wichtigste Traffic (C2-DNS und SMB) vollständig verpasst.

Was fehlt? — Defense-in-Depth

Eine Firewall allein reicht nicht. Die Cerber-Attacke zeigt, dass ein vollständiger Schutz mehrere Ebenen benötigt:

  1. DNS Security (Fortigate DNS Filter / Cisco Umbrella / Pi-hole) → C2-Domain-Blocking
  2. Internal Segmentation (Fortigate VLAN / NSX) → SMB-Lateral-Movement-Prevention
  3. IPS with SMB Signatures (MS17-010, EternalBlue) → Exploit-Detection
  4. Endpoint Detection (EDR) → Sysmon, Masquerading, Process-Monitoring
  5. Network Detection (NDR) → Zeek, Suricata auf internen Links
✅

Defense-in-Depth: Keine einzelne Sicherheitsmaßnahme ist ausreichend. Die Kombination aus Firewall, IDS, EDR und NDR deckt die Lücken, die jeweils andere hinterlassen. Genau das ist der Grund, warum ein SOC mehrere Datenquellen (Suricata, Sysmon, Registry, Firewall) korrelieren muss.

Fazit

Die Fortigate-Firewall in diesem Szenario erfüllte nur ihre Basis-Funktion (Broadcast-Denies). Die kritischen Bedrohungen — C2-Kommunikation und Lateral-Movement — passierten ungehindert.

Die Lehre für SOC-Teams:

  1. Firewall-Logs allein sind unzureichend — sie decken nur den Perimeter ab
  2. DNS-Filtering ist essenziell — ohne es ist C2-Kommunikation unsichtbar
  3. Internal Segmentation nötig — East-West-Traffic muss überwacht werden
  4. Multi-Source-Correlation — Suricata + Sysmon + Registry + Firewall = vollständiges Bild

Why Automate Hardening?

Manual hardening is error-prone and doesn't scale. A 20-device network means 20 opportunities to miss a step. When a new device is added, it starts with a blank security config. When an auditor asks for evidence, you have to check each device individually.

💡

Ansible solves all three problems. One playbook run hardens every device in the inventory simultaneously, new devices get the same config automatically on first run, and the YAML files are the evidence for compliance audits.

The key property is idempotency — running the playbook 10 times produces the same result as running it once. Ansible checks whether each setting is already correct before applying it, so re-runs are safe and fast.

Linux Hardening

The linux_hardening role covers three areas: SSH daemon configuration, UFW firewall rules, and kernel parameter hardening via sysctl.

🔐
SSH Hardening (CIS 5.2.x)

Deploys a Jinja2-templated sshd_config disabling root login, password auth, X11 forwarding, and weak ciphers. The template is validated with sshd -t before deployment — a broken config will never be written.

🔥
UFW Firewall (CIS 3.5.x)

Sets default policy to DENY incoming, then opens only explicitly listed ports. Management networks get a blanket ALLOW rule. Applied only on Debian/Ubuntu; Red Hat uses firewalld.

⚙️
Kernel Parameters (CIS 3.x)

Writes hardened sysctl settings to /etc/sysctl.d/99-cis-haertung.conf. Disables IP forwarding, activates reverse path filter, blocks ICMP redirects, enables SYN cookies against flood attacks.

Role Structure & Tag System

Each concern is a separate task file. The tag system lets you run only what you need:

roles/ ├── cisco_ios_hardening/ │ ├── tasks/ │ │ ├── main.yaml # orchestrator — include_tasks │ │ ├── ssh_haertung.yaml # tags: [ssh, banner, passwords] │ │ ├── aaa_konfiguration.yaml # tags: [aaa] │ │ ├── dienste_deaktivieren.yaml # tags: [services, ntp] │ │ ├── logging_konfiguration.yaml # tags: [logging] │ │ └── snmp_haertung.yaml # tags: [snmp] │ ├── defaults/main.yaml # all variables with safe defaults │ └── templates/login_banner.j2 # Jinja2 banner template │ └── linux_hardening/ ├── tasks/ │ ├── ssh_haertung.yaml # tags: [ssh, banner] │ ├── firewall_konfiguration.yaml # tags: [firewall] │ └── sysctl_haertung.yaml # tags: [sysctl] ├── defaults/main.yaml └── templates/ ├── sshd_config.j2 # hardened SSH config └── issue_banner.j2 # login warning banner
BASHCommon usage patterns
# Dry-run first — see what would change without touching anything ansible-playbook playbooks/site.yaml --check # Harden all Cisco devices ansible-playbook playbooks/cisco_hardening.yaml # Only apply SSH hardening across all platforms ansible-playbook playbooks/site.yaml --tags ssh # Only SNMPv3 on Cisco, with verbose output ansible-playbook playbooks/cisco_hardening.yaml --tags snmp -v
⚠️

Always run with --check first in production environments. The SSH hardening task includes a validate: parameter that runs sshd -t on the generated config before deploying — a syntax error will abort rather than lock you out.

Key Takeaways

01
YAML is the Audit Trail

The playbook files are human-readable documentation of exactly what security controls are applied and why. Version control provides a history of every change.

02
Tags Enable Surgical Runs

Running --tags ssh takes 30 seconds vs. 5 minutes for a full playbook. Critical for emergency remediation when you only need one fix applied immediately.

03
Validate Before Deploying

The sshd_config template uses Ansible's validate parameter to test the config with sshd -t before writing it. One extra line prevents lockouts.

04
Defaults Keep It Safe

Every variable has a sensible default in roles/*/defaults/main.yaml. The playbook works out of the box — customise only what you need to override.

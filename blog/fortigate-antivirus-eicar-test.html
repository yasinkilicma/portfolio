AntiVirus testen mit EICAR
— sauber durch, Malware geblockt

Nach dem IPS-Test der logische nächste Security-Profile-Test: greift der AntiVirus-Scan der FortiGate? Beweis mit der EICAR-Testdatei — dem harmlosen, aber von jeder AV-Engine erkannten Standard-Testmuster. Ein Client hinter der Firewall lädt sie über Klartext-HTTP: saubere Datei kommt durch, EICAR wird mitten im Transfer per Connection-Reset geblockt — inklusive AV-Log.

Aufbau

Der Client hängt hinter der FortiGate; sein HTTP-Verkehr läuft durch die LAN→WAN-Policy. Damit die AV-Engine den Inhalt sieht, muss es Klartext-HTTP sein (HTTPS bräuchte SSL-Inspektion). Die EICAR-Datei wird daher von einem kleinen HTTP-Server auf dem Lab-Host bereitgestellt.

Client (192.0.2.50, hinter der FW) │ GET http://<lab-host>/eicar.com ▼ FortiGate port3 (LAN 192.0.2.99) ──▶ Policy 1 (LAN→WAN, AntiVirus) ──▶ port1 │ ▼ AV scannt die HTTP-Antwort → EICAR erkannt → Connection Reset
💡

Warum EICAR: die EICAR-Datei ist kein echter Virus, sondern ein 68-Byte-Textmuster, das per Konvention von jeder AV-Engine als Treffer gemeldet wird. Sie steckt in der mitgelieferten AV-Datenbank — funktioniert also auch auf einer unlizenzierten Lab-VM, deterministisch.

1 — EICAR über HTTP bereitstellen

SHELL (Lab-Host)EICAR-Datei + HTTP-Server
# EICAR-Standardmuster (68 Byte) printf '%s' 'X5O!P%@AP[4\PZX54(P^)7CC)7}$EICAR-STANDARD-ANTIVIRUS-TEST-FILE!$H+H*' > www/eicar.com echo 'clean' > www/clean.txt python3 -m http.server 8000 --bind 0.0.0.0 # Klartext-HTTP

2 — AntiVirus an die Policy

Das eingebaute Profil default reicht — kein neues Objekt nötig (und schont das Eval-Lizenz-Limit). In der GUI: Policy & Objects → Firewall Policy → LAN-to-WAN → Security Profiles → AntiVirus = default. Per CLI:

FORTIOS CLIAV-Profil auf Policy 1
config firewall policy edit 1 set utm-status enable set av-profile "default" set ssl-ssh-profile "certificate-inspection" next end
🖱️

GUI statt flakiger CLI: Die serielle FortiGate-Konsole war bei längeren Sessions unzuverlässig — die Profilbindung per GUI vorzunehmen war robuster. Eine gute Erinnerung: das richtige Werkzeug für den Moment wählen.

3 — Der Test vom Client

CLIENT (hinter der FW)Saubere Datei vs. EICAR
# Kontrolle — saubere Datei curl -w '%{http_code}' http://<lab-host>/clean.txt -> 200 (kommt durch) # EICAR — wird gescannt curl http://<lab-host>/eicar.com -o /tmp/e \ -w 'code=%{http_code} size=%{size_download} exit=%{exitcode}' -> code=200 size=0 exit=56 (Recv failure: Connection reset by peer)

Der HTTP-Header (200) beginnt noch, doch sobald die AV-Engine im Body das EICAR-Muster erkennt, wird die Verbindung zurückgesetzt — der Client erhält 0 Byte. Die saubere Datei passiert unverändert. Klassisches Verhalten der flow-basierten AV.

Erkenntnisse (NSE4)

01
EICAR ist deterministisch

Der 68-Byte-Standard liegt in der mitgelieferten AV-DB — testet AV ohne echte Malware und ohne FortiGuard-Abo.

02
Klartext vs. HTTPS

Flow-AV sieht nur unverschlüsselten Inhalt. Für HTTPS bräuchte es SSL-Deep-Inspection (eigener Test).

03
Block = Connection Reset

Flow-basierte AV setzt bei einem Treffer die Verbindung zurück → Client bekommt 0 Byte (curl exit 56).

04
Eingebaute Profile nutzen

Das default-AV-Profil reicht und schont das Eval-Limit für eigene Objekte.

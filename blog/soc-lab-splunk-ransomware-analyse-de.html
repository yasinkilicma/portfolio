in einem Home-SOC-Lab detektieren

Wie ich eine vollständige SOC-Umgebung mit Terraform, Docker und Splunk Enterprise aufgebaut und anschließend einen echten Ransomware-Angriff anhand von 955.807 authentischen Angriffs-Events aus dem BOTS v1 Datensatz untersucht habe.

Lab-Architektur

Das Ziel war, eine realistische SOC-Umgebung von Grund auf per Infrastructure as Code aufzubauen — keine manuelle Konfiguration, alles reproduzierbar und versionskontrolliert auf GitHub.

Terraform (Infrastructure as Code) │ ├── docker_network → soc-lab-network (bridge) ├── docker_volume → soc-lab-splunk-data (persistent) └── docker_container → soc-lab-splunk │ ├── Port 8000 → Splunk Web-GUI ├── Port 8088 → HTTP Event Collector (HEC) └── Port 9997 → Universal Forwarder

Abb. 1 — Splunk SOC Security Dashboard: Top-Angriffs-Signaturen, Angreifer-IPs und Attack Timeline auf einen Blick

Splunk Enterprise läuft in Docker, vollständig von Terraform verwaltet. Das bedeutet: Die gesamte Lab-Umgebung kann mit einem einzigen terraform apply Befehl von Null aufgebaut werden.

Der Datensatz — BOTS v1

Der Boss of the SOC (BOTS) v1 Datensatz von Splunk enthält echte Angriffsdaten aus einer simulierten Unternehmensumgebung. Er deckt eine mehrstufige Angriffskampagne ab, die im August 2016 ein fiktives Unternehmen (Wayne Enterprises) angegriffen hat.

ℹ️

955.807 Events aus 22 Quell-Typen — darunter Suricata IDS, Windows Event Logs, FortiGate-Firewall, IIS-Webserver-Logs, Netzwerk-Streams (DNS, HTTP, SMB, TCP) und Nessus-Vulnerability-Scans.

Die Untersuchung

Schritt 1 — Reconnaissance: Was haben wir überhaupt?

Zunächst habe ich die verfügbaren Log-Quellen kartiert und identifiziert, welche Systeme angegriffen wurden.

SPL Alle Log-Quellen und Event-Anzahl im Überblick
# Events pro Quell-Typ zählen — um zu verstehen, welche Daten vorliegen index=botsv1 earliest=0 | stats count by sourcetype | sort -count

Ergebnis: Suricata IDS hatte 125.584 Alarme, SMB-Traffic 151.568 Events, Windows Security Logs 87.430 Einträge.

Schritt 2 — Zielsystem identifizieren

Nessus-Vulnerability-Scan-Logs zeigten, welche internen Systeme der Angreifer aktiv sondierte.

SPL Vulnerability-Scan-Ziele herausfinden
# Welche Hosts wurden gescannt? (Nessus = Angreifer kartiert das Netzwerk) index=botsv1 earliest=0 sourcetype=nessus:scan | stats count by host_ip | sort -count

Primärziel: 192.168.250.20 wurde 37-mal gescannt — eindeutig das Hauptangriffsziel.

Schritt 3 — IDS-Alarm-Analyse

Suricata-IDS-Alarme gegen die Ziel-IP zeigten die eingesetzten Angriffstechniken.

SPL Suricata-Alarme gegen das Hauptopfer
# Welche Angriffs-Signaturen haben gegen unser Ziel ausgelöst? index=botsv1 earliest=0 sourcetype=suricata dest_ip=192.168.250.20 | stats count by alert.signature | sort -count | head 10

🚨

Kritischer Fund: Die Signatur "ETPRO TROJAN Ransomware/Cerber Onion Domain Lookup" bestätigt, dass der Rechner 192.168.250.20 mit Cerber Ransomware infiziert war und aktiv mit einem Tor-basierten Command-&-Control-Server kommunizierte.

Schritt 4 — Angreifer-Zuordnung

Die Suche nach allen Quell-IPs zeigte sowohl den externen Angreifer als auch die laterale Ausbreitung im Netzwerk.

SPL Alle Angriffsquellen identifizieren
# Wer erzeugt den meisten bösartigen Traffic? index=botsv1 earliest=0 sourcetype=suricata | stats count by src_ip | sort -count | head 10
🔴
192.168.250.100 — Kompromittierter interner Rechner

2.907 Alarme — die höchste Anzahl aller Quellen. Ein interner Rechner führt laterale Bewegung durch: Die Infektion hat sich bereits im Netzwerk ausgebreitet.

⚠️
192.168.2.50 — Externer Angreifer

2.250 Alarme. Dies ist die IP-Adresse des ursprünglichen Bedrohungsakteurs — der Einstiegspunkt des Angriffs von außen.

ℹ️
8.8.8.8 — Google DNS

2.038 DNS-Anfragen an Googles öffentlichen DNS — typisches Verhalten des Domain Generation Algorithm (DGA) von Ransomware für die C2-Kommunikation.

Abb. 3 — Top Angreifer-IPs: 192.168.250.100 (interner kompromittierter Host) mit den meisten Alarmen — ein klares Zeichen für Lateral Movement

Rekonstruierte Angriffs-Timeline

Phase 1 — Reconnaissance

Externer Angreifer (192.168.2.50) führt Nessus-Vulnerability-Scans gegen interne Hosts durch. Ziel 192.168.250.20 als Hauptopfer identifiziert.

Phase 2 — Erstzugriff

Angreifer nutzt Schwachstelle auf 192.168.250.20 aus. SMB-Port-Scan (Port 445) und RPC-Sondierung (Port 135) deuten auf einen Windows-Exploit-Vektor hin.

Phase 3 — Cerber Ransomware

Cerber Ransomware wird auf 192.168.250.20 installiert. IDS erkennt Tor-(.onion)-Domain-Lookup — Ransomware kontaktiert seinen C2-Server für Verschlüsselungsschlüssel.

Phase 4 — Lateral Movement

Infektion breitet sich auf 192.168.250.100 aus. Dieser interne Rechner erzeugt das höchste Alarm-Volumen — 2.907 IDS-Alarme.

Phase 5 — Detektion

Suricata IDS erzeugt insgesamt 125.584 Alarme. SOC-Analyst korreliert Nessus-Scans, IDS-Alarme und Windows Event Logs zur vollständigen Angriffskette.

Wichtigste Erkenntnisse

01
IaC macht Labs reproduzierbar

Terraform + Docker bedeutet: Die gesamte SOC-Umgebung kann in Minuten von Null aufgebaut werden. Kein manuelles Setup, kein Configuration Drift.

02
Lateral Movement ist lautlos

Der kompromittierte interne Rechner (192.168.250.100) erzeugte mehr Alarme als der externe Angreifer — interner Traffic wird oft zu wenig überwacht.

03
Mehrere Quellen korrelieren

Keine einzelne Log-Quelle erzählte die ganze Geschichte. Die Angriffskette wurde erst sichtbar durch die Korrelation von Nessus-Scans, Suricata-Alarmen und Windows Event Logs.

04
DNS ist ein Warnsignal

2.038 DNS-Anfragen an 8.8.8.8 und fehlerhafte DNS-Antworten sind klassische C2-Kommunikationsmuster von Ransomware — DNS-Traffic immer überwachen.

🛡️ IPS NSE4 Juni 2026

IPS testen mit einer Custom-Signatur
— deterministisch statt Signatur-Lotterie

Wie beweist man, dass das Intrusion Prevention System einer FortiGate wirklich greift — ohne sich auf FortiGuard-Signaturen zu verlassen, die auf einer unlizenzierten Lab-VM fehlen können? Mit einer eigenen F-SBID-Signatur: ein eindeutiges Muster im HTTP-Verkehr eines Clients hinter der Firewall wird zuverlässig erkannt und geblockt. Kontroll-Request 200, Angriffs-Request dropped — inklusive IPS-Log.

Warum eine Custom-Signatur

Auf einer unlizenzierten Lab-FortiGate ist die FortiGuard-IPS-Datenbank nur eingeschränkt aktuell — sich für eine Demo auf eine bestimmte echte Signatur zu verlassen, ist eine Lotterie. Eine eigene Signatur ist deterministisch: sie feuert garantiert auf ein selbst definiertes Muster. Und „Custom IPS Signatures" sind ohnehin NSE4-Stoff.

🧪

Aufbau: Der Client hängt hinter der FortiGate (LAN-Segment via QEMU Socket-Networking). Sein HTTP-Verkehr läuft durch die LAN→WAN-Policy — genau dort greift die flow-basierte IPS-Inspektion.

1 — Die eigene Signatur

Eine F-SBID-Signatur, die das Muster NSE4IPSTEST in client-seitigem HTTP-Verkehr matcht:

FORTIOS CLIconfig ips custom
config ips custom edit "NSE4-Test" set signature "F-SBID( --name \"NSE4.IPS.Test\"; --protocol tcp; --service HTTP; --flow from_client; --pattern \"NSE4IPSTEST\"; --no_case; )" next end # FortiOS vergibt automatisch eine attack-id (hier: 6004)

2 — Signatur in einen Sensor, Sensor an die Policy

⚠️

Eval-Limit: Die kostenlose Lizenz deckelt die IPS-Sensor-Tabelle bei 10 — voll mit Default-Sensoren. Ein neuer Sensor scheitert mit Return code -4 (reached the maximum number of entries). Lösung: einen bestehenden Sensor wiederverwenden und einen Eintrag ergänzen.

FORTIOS CLISensor-Eintrag + Policy
# Eintrag referenziert die Signatur über die ID (set rule = Integer!) config ips sensor edit "Basic-IPS" config entries edit 10 set rule 6004 set status enable set action block set log enable set log-packet enable next end next end # auf die LAN->WAN-Policy anwenden config firewall policy edit 1 set utm-status enable set ips-sensor "Basic-IPS" set logtraffic all next end

3 — Der Test vom Client

Zwei Requests an dieselbe (Klartext-)HTTP-Seite — einmal sauber, einmal mit dem Muster in der URL:

CLIENT (hinter der FW)Kontrolle vs. Angriff
# Kontrolle — sauber curl -w '%{http_code}' http://neverssl.com/ -> 200 # Angriff — enthält das Signatur-Muster curl -w '%{http_code}' 'http://neverssl.com/x?p=NSE4IPSTEST' -> curl: (28) Connection timed out ... code=000 (von IPS verworfen)

Gleiche Ziel-URL, einziger Unterschied das Muster — die saubere Anfrage passiert, die „bösartige" wird verworfen. Da nur die IPS-Signatur das Muster inspiziert, ist der Block eindeutig ihr zuzuordnen.

Logging-Detail: Auf einer disklosen FortiGate-VM landen Logs im RAM. Tauchte zunächst kein IPS-Event auf, half set log enable am Sensor-Eintrag plus config log memory filter / set severity information. Der Block selbst funktioniert unabhängig davon — das Log ist die Sichtbarkeit.

Erkenntnisse (NSE4)

A
F-SBID-Syntax

--pattern, --service, --flow definieren eine eigene Signatur — deterministisch testbar.

B
set rule = ID

Im Sensor-Eintrag referenziert set rule die numerische Signatur-ID, nicht den Namen.

C
Flow-based IPS

IPS und Application Control sind flow-based; aktiviert über utm-status in der Policy.

D
Eval-Limits kennen

10 Sensoren, 3 Policies/Routen — Limits beim Design einplanen, vorhandene Objekte wiederverwenden.

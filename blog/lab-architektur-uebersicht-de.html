Home-Lab Architektur
FortiGate + Splunk SOC auf einem Mac

Wie man auf einem einzigen Mac mini M4 eine vollständige Netzwerk-Sicherheitsumgebung betreibt — mit einer echten Firewall-VM, einem Angreifer-Client und einem SIEM-System mit 955.807 realen Angriffsdaten.

Was ist dieses Lab?

Ein Home-Lab ist eine selbst aufgebaute Lernumgebung — kein Cloud-Dienst, kein Schulungsraum, sondern echte Software, die auf dem eigenen Gerät läuft. Dieses Lab besteht aus zwei Teilen:

FortiGate Firewall-Lab

Eine echte Unternehmens-Firewall läuft als virtuelle Maschine. Ein simulierter Angreifer-Client schickt Verkehr durch die Firewall — genau wie in einem echten Netzwerk.

Splunk SOC-Lab

Ein Security Operations Center (SOC) im Miniformat. Splunk sammelt alle Logs und zeigt in Dashboards, was im Netzwerk passiert — inklusive echter Ransomware-Angriffsdaten.

Beide Labs laufen gleichzeitig auf einem einzigen Mac mini M4. Keine Cloud-Kosten, keine separaten Server — alles virtualisiert auf einem Gerät.

Die einzelnen Komponenten erklärt

Was macht jede Komponente — und warum wurde sie so gewählt?

① Mac mini M4 — der Host

Der Mac mini M4 ist der physische Rechner, auf dem alles läuft. Er hat einen ARM64-Prozessor (Apple Silicon) — das bedeutet, alle virtuellen Maschinen müssen ebenfalls ARM64-Software verwenden. Das ist wichtig, weil ältere x86-VMs hier nicht laufen würden.

💡

Was ist ARM64? Eine Prozessor-Architektur — wie eine "Sprache", die der Chip versteht. Intel/AMD-PCs sprechen x86. Apple Silicon spricht ARM64. Software muss in der richtigen "Sprache" geschrieben sein, sonst läuft sie nicht.

② UTM 4.7.5 — der Hypervisor für die Firewall

UTM ist eine Virtualisierungssoftware für macOS — ähnlich wie VMware oder VirtualBox, aber speziell für Apple Silicon optimiert. UTM nutzt Apples eingebautes Hypervisor Framework (hvf), sodass die FortiGate-VM mit voller CPU-Geschwindigkeit läuft.

Hypervisor = VM-Gastgeber

Ein Hypervisor teilt die Hardware des Computers in mehrere virtuelle Computer auf. Jede VM denkt, sie hat echte Hardware — in Wirklichkeit teilt sie sich mit anderen VMs.

Apple hvf = Hardware-Beschleunigung

Ohne hvf würde jede CPU-Instruktion in Software emuliert — sehr langsam. Mit hvf laufen ARM64-VMs fast so schnell wie nativ auf der echten Hardware.

③ FortiGate-VM 7.4.12 ARM64 — die Firewall

FortiGate ist eine Enterprise-Firewall von Fortinet, die in Unternehmensnetzen weltweit eingesetzt wird. In diesem Lab läuft sie als virtuelle Maschine (VM) in UTM — mit denselben Features wie auf echter Hardware.

FortiGate-VM Netzwerk-Interfaces: port1 ──► UTM "Shared" (NAT) ──► Internet IP: 192.0.2.2 (WAN) Funktion: WAN-Anbindung + Management-GUI Mac-GUI: https://192.0.2.2 port3 ──► QEMU Socket-Netzwerk ──► Debian-Client IP: 192.0.2.99/24 (LAN) Funktion: internes Netzwerk für den Client DHCP-Server: .100 – .200
🔥

Was macht eine Firewall? Sie kontrolliert, welcher Netzwerk-Verkehr erlaubt ist. Jedes Paket, das vom Client ins Internet will, passiert die FortiGate — sie entscheidet: Durchlassen, Blockieren oder Loggen. Genau das passiert hier in Echtzeit.

④ QEMU Socket-Networking — das unsichtbare Kabel

Der Debian-Client läuft nicht in UTM, sondern in einem separaten QEMU-Prozess. Die Verbindung zwischen Client und FortiGate erfolgt über ein virtuelles Netzwerkkabel: QEMU Socket-Networking.

Wie QEMU Socket-Networking funktioniert: FortiGate (in UTM) Debian-Client (in QEMU) │ │ │ port3 listen=127.0.0.1:4700 │ connect=127.0.0.1:4700 │ └────────────────────────────────────┘ │ TCP-Socket (lokal) │ (nur innerhalb des Macs sichtbar) │ Start-Reihenfolge wichtig: 1. FortiGate starten → öffnet den Socket (wartet) 2. QEMU starten → verbindet sich mit dem Socket → Danach fließen Ethernet-Frames zwischen beiden VMs
💡

Warum kein UTM für den Client? UTM kann keine beliebigen QEMU-Argumente übergeben — es filtert Leerzeichen in den Parametern heraus, was die Socket-Konfiguration unmöglich macht. Der direkte QEMU-Befehl (brew install qemu) gibt volle Kontrolle.

⑤ Debian ARM64 (Kali-Client)

Der "Angreifer-Client" ist eine Debian ARM64 Cloud-Image VM. Kali Linux ist technisch gesehen Debian + Sicherheits-Tools — daher wurden einfach die gewünschten Tools nachinstalliert: nmap, hping3, tcpdump, curl.

Client-Konfiguration: IP-Adresse: 192.0.2.50/24 (statisch via Netplan) Gateway: 192.0.2.99 (FortiGate port3 = Default-GW) Konsole: TCP 127.0.0.1:4600 (seriell via nc oder expect) Start: ~/kali-lab/start-lab.sh Installierte Tools: nmap → Port-Scanner (Netzwerk-Erkennung) hping3 → Packet-Crafter (manuelle TCP/UDP-Pakete) tcpdump → Paket-Mitschnitt curl → HTTP-Client (für AV/IPS-Tests)

Das SOC-Lab: Splunk + BOTS v1

Parallel zum FortiGate-Lab läuft ein Security Operations Center (SOC) in Docker. Es besteht aus Splunk Enterprise und zwei Datenquellen.

⑥ Docker + Splunk Enterprise

SOC-Lab Komponenten: Mac mini M4 │ └── Docker Container: soc-lab-splunk │ ├── Port 8000 → Splunk Web-GUI (Browser: localhost:8000) ├── Port 8088 → HEC Endpoint (HTTP Event Collector) └── Port 9997 → Splunk Indexer (für Forwarder) Splunk Indexes (Datenspeicher): index=fortigate → Echtzeit-Logs aus dem FortiGate-Lab → Fließen via Python HEC-Bridge rein index=botsv1 → BOTS v1 Datensatz (955.807 Events) → Echter Ransomware-Angriff von 2016 → WICHTIG: lebt im Container (nie löschen!)
🏆

Was ist BOTS v1? "Boss of the SOC" ist ein öffentlicher Datensatz von Splunk — echte Logs eines simulierten Unternehmensangriffs (Wayne Enterprises, August 2016). Ein Angreifer installierte den Cerber-Ransomware-Trojaner, der Dateien verschlüsselte und sich seitlich im Netzwerk ausbreitete. Mit 955.807 Events ist das ein realistisches Übungs-Szenario für SOC-Analysten.

⑦ Python HEC-Bridge — Logs in Echtzeit

Die FortiGate sendet ihre Logs als Syslog UDP (Port 5514). Splunk versteht dieses Format nicht direkt — deswegen übersetzt ein Python-Skript die Logs und schickt sie über den HTTP Event Collector (HEC) an Splunk weiter.

Log-Pipeline (Echtzeit): FortiGate-VM │ │ Syslog UDP :5514 │ (jede Firewall-Aktion → ein Log-Eintrag) ▼ Python-Skript: fgt-syslog-hec.py │ • Empfängt UDP-Pakete auf Port 5514 │ • 4 Worker-Threads (parallele Verarbeitung) │ • Queue: max. 10.000 Einträge │ │ HTTPS POST → localhost:8088 │ Token: HEC_TOKEN (in config) ▼ Splunk HEC (HTTP Event Collector) │ │ sourcetype=fgt_log, index=fortigate ▼ Splunk Dashboard: FortiGate SOC (Traffic, IPS, AntiVirus, Port-Scans — alles live)

Was passiert bei einem Angriff — Schritt für Schritt

Beispiel: Der Kali-Client führt einen nmap-Port-Scan durch. So fließen die Daten:

1
Client startet den Scan
Der Kali-Client sendet TCP-SYN-Pakete an Ziel-IPs — ein normaler nmap-Port-Scan. Ziel: herausfinden, welche Ports offen sind.
2
Pakete treffen auf die FortiGate
Jedes Paket passiert die LAN→WAN-Policy. Die FortiGate analysiert den Traffic — Flow-Inspection, IPS, AntiVirus. Für jeden abgeschlossenen TCP-Flow entsteht ein Log-Eintrag.
3
Syslog → Python-Bridge → Splunk
Die FortiGate schickt den Log per UDP Syslog an Port 5514. Das Python-Skript empfängt ihn sofort und leitet ihn per HTTPS an Splunk HEC weiter. Latenz: unter 1 Sekunde.
4
Splunk indexiert und visualisiert
Splunk speichert den Event im Index fortigate. Das Port-Scan-Detection-Dashboard zeigt sofort: neue Scanner-IP, gescannte Ports, nmap-Fingerabdruck (sentbyte=164, rcvdbyte=60).

Lab starten — die richtige Reihenfolge

Die Startreihenfolge ist wichtig: Der QEMU-Socket-Listener muss vor dem Client geöffnet sein, sonst schlägt die Verbindung fehl.

Start-Reihenfolge: Schritt 1: Docker Splunk (läuft im Hintergrund, meist schon aktiv) docker start soc-lab-splunk Schritt 2: FortiGate-VM in UTM starten utmctl start E6133DD1-0297-4B83-9999-72BB1633FAB1 → Wartet auf Socket-Verbindung auf Port 4700 → Warten bis FortiOS vollständig gebootet ist (~30 Sek.) Schritt 3: Debian-Client (QEMU) starten ~/kali-lab/start-lab.sh → Verbindet sich mit FortiGate über Socket :4700 → Seriell-Konsole: nc 127.0.0.1 4600 Schritt 4: Python HEC-Bridge starten python3 ~/soc-lab/fgt-syslog-hec.py → Empfängt UDP 5514, leitet weiter an localhost:8088 Fertig: Alle Komponenten laufen — Splunk unter http://localhost:8000

Warum diese Architektur?

Jede Design-Entscheidung hat einen Grund. Hier die wichtigsten:

ENTSCHEIDUNGWARUM
FortiGate in UTM (nicht QEMU) UTM nutzt Apples Hypervisor Framework für native ARM64-Beschleunigung — keine Emulation, volle Performance.
Debian statt Kali Kali ist Debian. Der Kali-Installer scheiterte an fehlendem Framebuffer und UEFI-Problemen unter ARM64 hvf. Cloud-Image bootet direkt ohne Installer.
QEMU Socket statt zweites UTM-Interface UTM kann keine Leerzeichen in QEMU-Argumente einfügen — Socket-Parameter wären dadurch kaputt. Direktes QEMU gibt volle Kontrolle.
Python HEC-Bridge statt Syslog-Forwarder Splunk's nativer Syslog-Input erfordert einen Universal Forwarder oder Add-on. Die Python-Bridge ist ein einziges Skript, keine Installation nötig.
BOTS v1 im Container (nie löschen) Die 955.807 Events liegen in der Writable-Layer des Containers. Docker-Volume-Migration oder Neuanlage würde alle Daten verlieren.

Übung 1 — CLI Temel Komutlar: get vs show vs diagnose

Das wichtigste Unterscheidungsmerkmal in der NSE4: get zeigt den aktuellen running state, show die gespeicherte Config (VDOM-bezogen) und diagnose ist für Troubleshooting.

CLI System-Status und Performance abfragen
get system status get system performance status get system interface get system arp get router info routing-table all execute ping 8.8.8.8
💡

NSE4 Merke: get zeigt immer den aktuellen Zustand — inklusive per DHCP bezogener IP, Lease-Zeit, Gateway und MAC-Adresse. show zeigt nur das, was der Admin explizit konfiguriert hat.

Die wichtigsten Erkenntnisse

01
get = Running State

DHCP-IP, Lease-Zeit, Gateway, MAC, Link-Status — alles nur über get sichtbar.

02
show = Config

Zeigt nur explizit gesetzte Werte. Default-Werte (z.B. nat disable) werden ausgeblendet.

03
Distance-Werte

DHCP-Default-Route hat Distance 5, statische Route 10 — ein häufiger NSE4-Prüfungsfall.

04
Tunnel-Interfaces

naf.root, l2t.root, ssl.root sind Standard-Tunnel — aktiv auch ohne VPN-Config.

Übung 2 — Interface-Verwaltung & DHCP

FortiGate unterscheidet zwischen physischen Interfaces, Aggregations und Tunneln. Im Lab sind port1 (DHCP/WAN) und port2 (statisch/LAN) aktiv. Zusätzlich existiert fortilink für FortiSwitch-Auto-Discovery.

CLI Interface-Config im Kontext
config system interface edit port1 show # Config get # Running State next edit port2 show next end show system dhcp server get system dns
✅

Praktischer Nebeneffekt: fortilink hat die IP 10.255.1.1/24 und ist standardmäßig aktiv. Damit können FortiSwitch- Geräte per Plug-and-Play erkannt werden (VCI-Match auf FortiSwitch und FortiExtender).

DHCP-Server am LAN-Interface

Der DHCP-Server ist an port2 gebunden und verteilt IPs aus dem Bereich .100 bis .200. Als Default-Gateway fungiert die FortiGate selbst (192.168.x.x).

DHCP-SERVER Config auf port2
config system dhcp server edit 1 set default-gateway 192.168.x.x set netmask 255.255.255.0 set interface "port2" config ip-range edit 1 set start-ip 192.168.x.100 set end-ip 192.168.x.200 next end set vci-match enable set vci-string "FortiSwitch" "FortiExtender" next end

Übung 3 — Firewall-Policies: NAT, Sequenz & Lizenz-Limits

Firewall-Policies werden top-down abgearbeitet. Die Evaluation-Lizenz erlaubt maximal 3 Policies — ein harter, aber realistischer Limit für das Lab.

POLICY Bestehende Policy anzeigen und verstehen
show firewall policy # Ausgabe (bereits konfiguriert) config firewall policy edit 1 set name "LAN-to-WAN" set srcintf "port2" set dstintf "port1" set action accept set srcaddr "all" set dstaddr "all" set schedule "always" set service "ALL" set nat enable # Source NAT next end

Neue Policy hinzufügen (Evaluation: max 3)

Wir haben eine zweite Policy Management-HTTPS für WAN-Zugriff auf die GUI erstellt und anschließend eine dritte Test-Policy.

CLI 2. Policy erstellen
config firewall policy edit 0 # 0 = auto ID set name "Management-HTTPS" set srcintf "port1" set dstintf "port1" set action accept set service "HTTPS" set nat disable set comments "WAN management access" next end
🚫

Der Lizenz-Limit-Test: Beim Versuch, eine 4. Policy zu erstellen, liefert die CLI: return code -4 (reached the maximum number of entries). Genau das ist ein typischer NSE4-Prüfungsfall.

FEHLER Lizenz-Limit erreicht
edit 0 Command fail. Return code -4 (reached the maximum number of entries)

Policy-Sequenz ändern

Die Reihenfolge ist entscheidend: FortiGate arbeitet Policies von oben nach unten ab. Um eine Policy zu verschieben:

CLI Policy-Sequenz anpassen
config firewall policy move 2 before 1 # Policy 2 vor Policy 1 move 1 after 3 # Policy 1 nach Policy 3 end

Echte Lab-Ausgabe — das Policy-Limit live

Der Versuch, eine vierte Policy anzulegen, scheitert sofort am Eval-Lizenz-Limit — Fehlercode -4:

ECHTE LAB-AUSGABE 4. Policy → Limit erreicht
FGT-LAB # config firewall policy FGT-LAB (policy) # edit 0 Command fail. Return code -4 (reached the maximum number of entries) FGT-LAB (policy) # set name "Exceed-Limit" Unknown action 0

Übung 4 — Routing-Tabelle & Statische Routen

Die Routing-Tabelle zeigt, wie FortiGate Entscheidungen trifft. Jede Route hat eine Administrative Distance — je niedriger, desto vertrauenswürdiger.

ROUTING Aktuelle Routing-Tabelle anzeigen
get router info routing-table all # Ausgabe Routing table for VRF=0 S* 0.0.0.0/0 [5/0] via 192.168.x.x, port1, [1/0] C 192.168.x.0/24 is directly connected, port2 C 192.168.x.0/24 is directly connected, port1 # Legende S* = Static (candidate default) C = Connected
📊

Administrative Distance (NSE4!): Connected = 0, DHCP-Route = 5, Static = 10, eBGP = 20 / iBGP = 200, OSPF = 110. Die DHCP-Default-Route mit Distance 5 übertrumpft eine statische Route mit 10.

Statische Route hinzufügen

CLI Dummy-Route für 10.0.0.0/8
config router static edit 0 set dst 10.0.0.0 255.0.0.0 set device port1 set gateway 192.168.x.x set distance 10 next end get router info routing-table all # 10.0.0.0/8 [10/0] via ... erscheint als S

Echte Lab-Ausgabe — Administrative Distance in der Tabelle

Nach dem Anlegen einer statischen Route zu 10.0.0.0/8 — beachte die Werte in eckigen Klammern [Distance/Metrik]:

ECHTE LAB-AUSGABE get router info routing-table all
FGT-LAB # get router info routing-table all ... Routing table for VRF=0 S* 0.0.0.0/0 [5/0] via 192.168.64.1, port1 # DHCP-Default → Distance 5 S 10.0.0.0/8 [10/0] via 192.168.64.1, port1 # statisch → Distance 10 C 192.168.1.0/24 is directly connected, port2 C 192.168.64.0/24 is directly connected, port1

Übung 5 — Security Profiles: IPS, Flow vs. Proxy

Security Profiles sind das Herzstück der FortiGate-Security. Der entscheidende NSE4-Unterschied: Flow-based vs. Proxy-based Inspection.

IPS Sensor erstellen und zuweisen
config ips sensor edit "Basic-IPS" set comment "NSE4 Practice" config entries edit 1 set severity medium high critical set action block next end next end # Der Policy zuweisen config firewall policy edit 1 set utm-status enable set ips-sensor "Basic-IPS" next end
✅

Erfolg: Nach set utm-status enable und set ips-sensor "Basic-IPS" zeigt die Policy utm-status: enable. Ein Warning "non-inspection profile doesn't perform SSL inspection" ist normal und kein Fehler.

Flow-based vs. Proxy-based Inspection

PROFILEMODENSE4-RELEVANZ
IPSFlow-basedSchnell, Paket-basiert, geringe Latenz
Application ControlFlow-basedApp-ID, QoS, Traffic Shaping
AntivirusFlow / ProxyBeide Modi; Proxy = Datei-Rekonstruktion, tiefer
Web FilterFlow / ProxyBeide Modi; Proxy bietet mehr URL-/SSL-Optionen
DLPFlow / ProxyBeide Modi; Proxy = Fingerprinting/Content
📝

NSE4 Merke: IPS und Application Control sind immer flow-based. AntiVirus, Web Filter und DLP unterstützen beide Modi — gesteuert per Policy (set inspection-mode flow|proxy). Proxy bietet tiefere Analyse (Datei-Rekonstruktion, mehr SSL-Optionen), Flow geringere Latenz.

Echte Lab-Ausgabe — IPS-Sensor an die Policy binden

Beim Aktivieren von utm-status + IPS auf Policy 1 warnt FortiOS, dass ohne SSL-Inspection-Profil keine SSL-Inspektion erfolgt:

ECHTE LAB-AUSGABE utm-status + ips-sensor + Warnung
FGT-LAB (1) # set utm-status enable FGT-LAB (1) # set ips-sensor "Basic-IPS" FGT-LAB (1) # next This policy has the following warnings: The non-inspection profile doesn't perform SSL inspection. # in der Policy bestätigt (get): inspection-mode : flow utm-status : enable ips-sensor : Basic-IPS ssl-ssh-profile : no-inspection

Übung 6 — Log & Debug: Troubleshooting

Debug ist die wichtigste Waffe für jeden NSE4-Kandidaten. Mit diagnose debug lässt sich der Paketfluss live verfolgen — ideal, um NAT-, Routing- oder Session-Probleme zu finden.

DEBUG Paket-Trace für einen bestimmten Host
diagnose debug enable diagnose debug flow filter addr 192.168.x.x diagnose debug flow trace start 10 # Einen Ping auslösen execute ping 192.168.x.x diagnose debug flow trace stop diagnose debug disable diagnose debug reset

Debug-Output lesen

Der Debug-Output zeigt pro Zeile einen Paket-Durchlauf:

DEBUG-LOG Ausschnitt aus dem Paket-Trace
id=xxxxx trace_id=1 func=print_pkt_detail msg="vd-root:0 received a packet(proto=6, 192.168.x.x:443->192.168.x.x:xxxxx) tun_id=0.0.0.0 from port1." id=xxxxx trace_id=1 func=resolve_ip_tuple_fast msg="Find an existing session, id-xxxxxx, original direction" # Analyse - proto=6 → TCP - from port1 → Eingehend auf WAN-Interface - session found → Session-basierte Firewall
01
Filter setzen

Ohne filter addr produziert Debug zu viel Output. Immer zuerst filtern.

02
Session-ID merken

session id-xxxxxx identifiziert den Flow. Mit diagnose sys session list nachschlagen.

03
Richtung erkennen

original direction = Client → Server, reply direction = Server → Client.

04
Debug sauber beenden

Immer disable und reset am Ende. Aktives Debug belastet die CPU.

Übung 7 — VPN: IPsec Site-to-Site

VPN ist ein Pflichtthema in der NSE4. Die Konfiguration erfolgt in zwei Phasen: Phase 1 (IKE — Authentication & Key Exchange) und Phase 2 (IPsec — Datenverschlüsselung).

VPN PHASE 1 IKE-Interface konfigurieren
config vpn ipsec phase1-interface edit "VPN-to-Lab2" set interface "port1" set ike-version 2 set dhgrp 14 set remote-gw 203.0.113.1 set psksecret "StrongPresharedKey123!" next end
VPN PHASE 2 IPsec-Selector erstellen
config vpn ipsec phase2-interface edit "VPN-to-Lab2-p2" set phase1name "VPN-to-Lab2" set dhgrp 14 set keylifeseconds 3600 next end

VPN-Tunnel überwachen

DIAGNOSE Tunnel-Status prüfen
diagnose vpn tunnel list # Ausgabe name=VPN-to-Lab2 ver=2 serial=1 192.168.x.x:0->203.0.113.1:0 status=down dst_mtu=0 weight=1 mode=auto/1 encap=none dpd: mode=on-demand status=fail # status=down ist normal — remote GW ist nicht erreichbar # dpd: Dead Peer Detection versucht, den Peer zu erreichen
🔐

NSE4 Merke: Phase 1 = IKE (Authentication, DH Group, Key Exchange). Phase 2 = IPsec (ESP/AH, Traffic Selector, SA). ike-version 2 ist moderner und sicherer als IKEv1. dhgrp 14 = 2048-bit MODP Group.

Echte Lab-Ausgabe — Tunnel-Status prüfen

Ohne konfigurierte Gegenstelle bleibt der Tunnel erwartungsgemäß status=down — Phase 1 und Phase 2 sind aber sauber angelegt:

ECHTE LAB-AUSGABE diagnose vpn tunnel list
FGT-LAB # diagnose vpn tunnel list list all ipsec tunnel in vd 0 ------------------------------------------------------ name=VPN-to-Lab2 ver=2 serial=1 192.168.64.2:0->203.0.113.1:0 nexthop=192.168.64.1 status=down role=primary accept_traffic=1 proxyid=VPN-to-Lab2-p2 proto=0 sa=0 ref=1 serial=1 stat: rxp=0 txp=0 rxb=0 txb=0 dpd: mode=on-demand status=fail retry=3 count=0

Übung 8 — NSE4 Quiz & Prüfungs-Strategie

Die folgenden Fragen basieren auf den Erfahrungen aus den vorherigen 7 Übungen. Sie decken die häufigsten NSE4-Themen ab.

1. Welcher Unterschied besteht zwischen get und show?
A) get = running state/hardware, show = config (VDOM-basiert)
B) get = hardware info, show = software config
C) get = partial config, show = full config
D) get = config only, show = statistics
Antwort A. get zeigt den aktuellen Zustand inklusive DHCP-IPs, MAC-Adressen und Link-Status. show zeigt nur die gespeicherte Konfiguration des aktuellen VDOM.
2. Wie viele Firewall-Policies sind unter einer Evaluation-Lizenz maximal erlaubt?
A) 10
B) 3
C) Unbegrenzt
D) 1
Antwort B. Evaluation-Lizenz = 1 vCPU, 2 GB RAM, 3 Interfaces, 3 Policies, 3 Routen. Der Fehler return code -4 erscheint beim Überschreiten.
3. In welchem Inspection-Modus läuft ein IPS-Sensor?
A) Proxy-based
B) Flow-based
C) Passive
D) Inline
Antwort B. IPS und Application Control arbeiten immer flow-based (schnell, paket-basiert). AntiVirus, Web Filter und DLP können dagegen flow- oder proxy-based laufen (Proxy = tiefere Inspektion) — der Modus wird in der Policy festgelegt.
4. Welches Kommando verschiebt Policy 2 vor Policy 1?
A) move 2 before 1
B) move 1 after 2
C) set sequence 1
D) reorder 2 1
Antwort A. move 2 before 1 ist die korrekte Syntax. Policies werden top-down abgearbeitet — die Reihenfolge ist entscheidend.
5. Was ist der Default-Wert der Administrative Distance für eine statische Route?
A) 1
B) 5
C) 10
D) 20
Antwort C. Static = 10. Die DHCP-Default-Route hat Distance 5 und übertrumpft somit eine statische Default-Route mit 10. eBGP = 20 / iBGP = 200, OSPF = 110.
6. Was verwaltet VPN Phase 1 (IKE)?
A) Datenverschlüsselung
B) IKE-Authentifizierung & Key Exchange
C) Routing-Tabelle
D) NAT
Antwort B. Phase 1 = IKE (Authentication, DH Group, Preshared Key/RSA). Phase 2 = IPsec (eigentliche Datenverschlüsselung über ESP/AH).
7. Was ist der Hauptunterschied zwischen NAT Mode und Transparent Mode?
A) NAT Mode ist schneller
B) Transparent Mode arbeitet als Layer-2-Bridge ohne IP
C) Transparent Mode ist nur für Wireless
D) NAT Mode unterstützt nur IPv6
Antwort B. Transparent Mode = Layer-2-Bridge, keine eigene IP (außer Management-IP), kein Routing, kein NAT. NAT Mode = Layer-3, Routing, NAT, Source/Destination NAT.

Quiz-Ergebnisse

Ein perfekter NSE4-Kandidat sollte mindestens 6 von 7 Fragen richtig beantworten. Die häufigsten Fehlerquellen: Administrative Distance (5 vs. 10) und der Unterschied zwischen NAT und Transparent Mode.

Kontext — zwei Labs, ein Problem

Das Heimlab besteht aus zwei unabhängigen Teilen: einer FortiGate-VM (UTM/QEMU auf dem Mac mini M4) und einem Splunk-Container (Terraform + Docker), in dem 955.807 Events aus dem BOTS v1-Datensatz liegen — darunter der gesamte Cerber-Ransomware-Angriff.

Das Ziel: FortiGate-Syslog in Echtzeit in Splunk einspeisen und dort dashboarden. Das Problem: Der BOTS-Datensatz (botsv1_data_set, 322 MB) lebt in der beschreibbaren Container-Schicht — nicht im persistenten Volume. Den Container neu zu erzeugen (z. B. via terraform apply mit neuen Ports) würde 955.807 Events vernichten. Also kein Container-Rebuild.

💡

Lösung: Splunk hat bereits Port 8088 (HEC) offen — HTTP Event Collector. Statt den Container anzufassen, baut eine schlanke Python-Bridge eine Brücke: FortiGate → UDP-Syslog → Bridge → HTTPS-HEC → Splunk. Kein Rebuild, keine Datenverlust-Gefahr.

Architektur

FortiGate-VM (UTM / QEMU, FortiOS 7.4.12 ARM64) port1 ── 198.51.100.2 (WAN / vmnet-shared) port3 ── 192.0.2.99 (LAN) │ │ syslog UDP → 198.51.100.1:5514 ▼ Python-Bridge (~/soc-lab/fgt-syslog-hec.py, 4 Worker-Threads) │ │ HTTPS POST → localhost:8088/services/collector/event ▼ Splunk-Container (soc-lab-splunk, Docker, Splunk Enterprise) index fortigate · sourcetype fgt_log │ ▼ SOC-Dashboard http://localhost:8000/en-US/app/search/fortigate_soc ── Kali-Client ────────────────────────────────────────────────────── brew-qemu (Debian 12 ARM64 cloud image) 192.0.2.50 netdev socket → FortiGate port3 (L2-Segment, inspiziert)

1 — Splunk vorbereiten: Index + HEC-Token

Zuerst einen eigenen Index fortigate anlegen und einen HEC-Token mit passendem Sourcetype erstellen — alles per Splunk-CLI innerhalb des Containers, da Port 8089 (Management-REST) nicht nach außen gemappt ist.

SHELLIndex + HEC-Token anlegen
# Index anlegen docker exec -u splunk soc-lab-splunk \ /opt/splunk/bin/splunk add index fortigate -auth admin:SOClab123! # HEC-Token erstellen (sourcetype + Ziel-Index fix verdrahtet) docker exec -u splunk soc-lab-splunk \ /opt/splunk/bin/splunk http-event-collector create fortigate-hec \ -description "FortiGate syslog bridge" \ -index fortigate \ -sourcetype fgt_log \ -uri https://localhost:8089 \ -auth admin:SOClab123! -> token=bd4870ce-a6a8-48e9-8b23-d6a02a7029ae

2 — Die Python-HEC-Bridge

Ein schlankes Python-Script wartet auf UDP-Datagramme (FortiGate-Syslog) und leitet jeden Datensatz als JSON-Event an den Splunk HEC weiter — mit 4 parallelen Worker-Threads, damit kein Burst-Traffic verloren geht.

PYTHONfgt-syslog-hec.py (Kernlogik)
import socket, ssl, json, time, urllib.request, threading, queue LISTEN_PORT = 5514 HEC_URL = "https://127.0.0.1:8088/services/collector/event" HEC_TOKEN = "<dein-token>" INDEX = "fortigate" SOURCETYPE = "fgt_log" _ctx = ssl.create_default_context() _ctx.check_hostname = False _ctx.verify_mode = ssl.CERT_NONE # self-signed Splunk-Zertifikat def sender(q): while True: raw, addr = q.get() payload = json.dumps({ "event": raw.decode("utf-8", "replace").strip(), "sourcetype": SOURCETYPE, "index": INDEX, "host": addr[0], "time": time.time(), }).encode() req = urllib.request.Request( HEC_URL, data=payload, headers={"Authorization": "Splunk " + HEC_TOKEN}) urllib.request.urlopen(req, context=_ctx, timeout=10) q.task_done()

Die Bridge läuft im Hintergrund (nohup python3 fgt-syslog-hec.py &) und ist in ~/kali-lab/start-lab.sh integriert — nach jedem Neustart des Mac mini startet sie automatisch mit dem Rest des Labs.

3b — Port 8080 für AV-Inspektion freischalten

Das eingebaute default-Protocol-Options-Profil ist schreibgeschützt und inspiziert HTTP nur auf Port 80. Das Lab-HTTP-Server läuft auf 8080 (Port 80 ist von Docker belegt) — also ein eigenes Profil:

FORTIOS CLINeues Protokoll-Profil + Policy
# Eigenes Protokoll-Profil: HTTP auf 80 + 8080 config firewall profile-protocol-options edit "lab-http" set comment "lab: HTTP on 80+8080" config http set ports 80 8080 set status enable end next end # Policy 1 (LAN→WAN) auf das neue Profil umstellen config firewall policy edit 1 set profile-protocol-options "lab-http" next end
📌

Warum ein eigenes Profil nötig war: FortiOS schützt default-Objekte vor Änderungen (read-only factory default). Port 8080 ist kein Standard-HTTP-Port — FortiGate hätte den Traffic durchgelassen, ohne ihn zu scannen. Das neue Profil lab-http löst das mit zwei Zeilen.

5 — Tests vom Kali-Client

Der Kali-Client (Debian 12 ARM64 hinter der FortiGate) schickt echten Angriffsverkehr durch die Firewall. Jede Verbindung läuft durch die LAN→WAN-Policy mit AV-, IPS- und App-Control-Profil — und landet Sekunden später im Splunk-Dashboard.

Test 1 — EICAR (AntiVirus)

KALI CLIENTSaubere Datei vs. EICAR-Testdatei
# Kontrolle: saubere Datei → kommt durch curl -s http://198.51.100.1:8080/clean.txt -> "clean file ok" (HTTP 200) # EICAR: wird von FortiGate flow-AV geblockt curl -s --max-time 15 \ -w 'HTTP_%{http_code}_BYTES_%{size_download}' \ -o /dev/null \ http://198.51.100.1:8080/eicar.com -> HTTP_200_BYTES_0 ← 200er Header kam, Body = 0 Byte (Connection Reset)

Die FortiGate beginnt, den HTTP-Response-Header weiterzuleiten (daher HTTP 200), erkennt im Body das EICAR-Muster und bricht die Verbindung ab — der Client erhält exakt 0 Byte. Klassisches Verhalten der flow-basierten AV-Inspektion.

Erkenntnisse

01
HEC statt Container-Rebuild

Wertvolle Daten im Container-Layer? Nie den Container anfassen. HEC ist die chirurgische Alternative — neue Datenquelle, bestehende Daten unangetastet.

02
Port-Inspektion explizit konfigurieren

FortiGate scannt HTTP nur auf Port 80. Non-Standard-Ports (8080) brauchen ein eigenes Protocol-Options-Profil — sonst passiert Traffic ungeprüft.

03
flow-AV: 200 + 0 Byte

Flow-basierte AV startet die Antwort, bevor der vollständige Body da ist. Erkennt sie das Muster, bricht sie die Verbindung ab — HTTP-Status-Code ist kein Indikator für Erfolg.

04
FortiOS-Felder in SPL direkt nutzbar

key=value-Syslog + Splunk KV-Extraktion = sofort suchbare Felder (srcip, action, virus, attack) ohne TA-Installation.

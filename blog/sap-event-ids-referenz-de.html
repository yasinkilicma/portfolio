SAP vs. Windows Event IDs: Windows verwendet numerische Event IDs (4625, 4688…). SAP verwendet alphanumerische Message IDs im Security Audit Log (SM20/RSAU_READ_LOG). Format: 2 Buchstaben + 1-3 Zeichen — z.B. AU2 (fehlgeschlagener Login), AUB (Benutzeränderung), EU1 (Tabellenzugriff).

01 — Authentication & Logon

Anmeldeereignisse sind das Fundament jeder SAP-Sicherheitsanalyse. Brute-Force erkennst du mit AU2 → AUI, kompromittierte Konten mit AU1 nach AUI (Konto entsperrt, dann sofort Login).

Message ID Beschreibung Angriff / Szenario Priorität MITRE
AU1 Erfolgreicher Dialog-Login
GUI / Web-Login erfolgreich — Baseline-Ereignis 		After-Hours Login, unbekannte IP INFO T1078
AU2 Fehlgeschlagener Login
Falsches Passwort, falscher User — Zähler für Lockout 		Brute-Force, Password Spray CRITICAL T1110
AU3 Erfolgreicher RFC-Login
Programmatischer Zugang — Batch, externe Systeme, Angreifer-Tools 		RFC Exploitation, externe Verbindung MEDIUM T1133
AU4 Fehlgeschlagener RFC-Login
Externes System versucht RFC-Verbindung — oft von Angriffstools 		Externer RFC-Angriff, Tool-Scanning CRITICAL T1110
AU5 Logoff
Session-Ende — Session-Dauer berechenbar 		Sehr kurze Sessions = automatische Angriffs-Tools INFO —
AUI Benutzer gesperrt (Lockout)
Nach Überschreiten max. Fehlversuche — automatisch gesperrt 		Brute-Force-Bestätigung, DoS auf User CRITICAL T1110
SPL Brute-Force Erkennung — AU2 Häufung → AUI Lockout
// Schritt 1: Fehlversuche zählen index=sap_audit msgid="AU2" | bucket _time span=5m | stats count as failed_logins by _time, user, terminal | where failed_logins >= 5 | sort - failed_logins // Schritt 2: Lockout bestätigen index=sap_audit (msgid="AU2" OR msgid="AUI") | eval event_type=if(msgid="AU2","Fehlversuch","GESPERRT") | table _time, event_type, user, terminal, client | sort _time
SPL Kompromittiertes Admin-Konto — After-Hours Login mit sensitiven T-Codes
index=sap_audit msgid="AU1" | eval hour=strftime(_time,"%H") | where (hour<6 OR hour>=22) | join user [ search index=sap_audit msgid="AU6" match(transaction, "(?i)(SE38|SU01|PFCG|SM59|SM30)") | stats values(transaction) as sensitive_tcodes by user ] | table _time, user, terminal, sensitive_tcodes

02 — User & Role Management

Änderungen an Benutzern und Rollen ohne Change-Management-Ticket sind sofort verdächtig. AUB ist der wichtigste Indikator für Privilege Escalation im SAP-Umfeld.

Message ID Beschreibung Angriff / Szenario Priorität MITRE
AUA Benutzer erstellt
Neues SAP-Konto angelegt — wer hat erstellt, wann, für wen 		Backdoor-Konto, Persistence CRITICAL T1136
AUB Benutzer geändert (Rollen/Profile)
Rolle oder Profil zugewiesen/entfernt — AUB-Storm = viele AUF + AUB 		Privilege Escalation — kritischstes SAP-Event CRITICAL T1078.002
AUC Benutzer gelöscht
Konto entfernt — Spurenbeseitigung nach Backdoor-Nutzung 		Cover Tracks HIGH T1070
AUD Benutzer gesperrt/entsperrt (Admin)
Explizites Sperren oder Entsperren durch Administrator 		Admin-Manipulation, Dormant Account HIGH T1078
AUE Passwort geändert
Passwortreset — nach Kompromittierung oft sofort geändert 		Account Takeover, Post-Kompromittierung HIGH T1098
!

AUB Storm: Wenn ein Benutzer innerhalb kurzer Zeit viele AUF-Ereignisse (Authorization Failures) produziert — auf verschiedenen T-Codes wie PFCG, SE11, SM30, SM59, SU01 — spricht man von einem AUB Storm. Das ist ein klares Zeichen für systematische Privilege Escalation.

SPL AUB Storm Detection — Privilege Escalation via Authorization Failures
index=sap_audit msgid="AUF" | bucket _time span=15m | stats count as auth_failures, dc(object) as unique_tcodes, values(object) as tcodes_list by _time, user | where auth_failures >= 10 AND unique_tcodes >= 3 | sort - auth_failures

03 — Transaction & Authorization Access

T-Code-Zugriffe zeigen, was ein Benutzer im System tut. Bestimmte T-Codes — SU01, PFCG, SE38, SM59 — sind hochprivilegiert und sollten eng überwacht werden.

Message ID Beschreibung Angriff / Szenario Priorität MITRE
AU6 Transaction gestartet
Welcher T-Code wurde aufgerufen — SE38, SU01, PFCG, SM59, SM30 		Priviligierter T-Code-Zugriff, Backdoor-Admin HIGH T1059
AUF Authorization Check fehlgeschlagen
Zugriff verweigert — Kernstück des AUB-Storm-Musters 		Privilege Escalation Versuch, AUB Storm CRITICAL T1078
AUG Systemoption geändert
System Change Option oder Mandanteneinstellung modifiziert 		Defense Evasion, System Manipulation HIGH T1562
SPL Kritische T-Code Überwachung — SU01, PFCG, SE38, SM59
index=sap_audit msgid="AU6" | where match(transaction, "(?i)^(SU01|PFCG|SE38|SM59|SM30|SE11|SM50|AL08|SUIM)$") | stats count by user, transaction, terminal | sort - count

04 — RFC & Remote Access

RFC (Remote Function Call) ist ein SAP-eigenes Protokoll für programmtische Fernzugriffe. Angreifer nutzen RFC, um mit Tools wie RFC_READ_TABLE direkt Tabellen auszulesen — ohne SAP-GUI-Login.

Message ID Beschreibung Angriff Priorität MITRE
AU3 Erfolgreicher RFC-Login
Programmatischer Zugang erfolgreich — von welcher IP? 		Legitimer Batch oder Angreifer-Tool MEDIUM T1133
AU4 Fehlgeschlagener RFC-Login
Externe IP versucht RFC-Verbindung — oft mit Default-Usernamen 		RFC Brute-Force, Scanning mit Angriffs-Tools CRITICAL T1110
AU9 RFC-Funktionsaufruf
Welcher Funktionsbaustein wurde per RFC aufgerufen 		RFC_READ_TABLE, RFCPING, SAPMSSY1 = Angriffs-Tools HIGH T1059
⚠

Bekannte Angriffs-Funktionsbausteine: RFC_READ_TABLE (Tabellen auslesen), RFCPING (Erreichbarkeit testen), SAPMSSY1 (System-Info), TH_WPINFO (Work-Process-Info). Diese werden von SAP-Exploit-Tools wie SAP Metasploit-Modulen verwendet.

SPL Externer RFC-Angriff — fehlgeschlagene RFC-Logins von externen IPs
index=sap_audit msgid="AU4" | eval is_external = if( NOT match(terminal, "^(10\.|192\.168\.|172\.)"), "EXTERN", "intern" ) | where is_external="EXTERN" | stats count as failed_rfc by user, terminal | sort - failed_rfc

05 — Datenexfiltration (Table Access)

SAP-Datenexfiltration erfolgt meist über direkte Tabellenzugriffe mit EU1. Das Muster: viele Datensätze aus sensiblen Tabellen innerhalb von Sekunden — mit RFC_READ_TABLE oder direkt über SE16/SE16N.

Message ID Beschreibung Kritische Tabellen Priorität MITRE
EU1 Tabellen-Lesezugriff (SELECT)
Wie viele Datensätze? In welcher Zeit? Von wem? 		USR02, AGR_USERS, RFCDES, T000, ABAP_SOURCE CRITICAL T1005
EU2 Generischer Tabellenzugriff
Tabellenänderungen über SM30/SE16 — Manipulation von Stammdaten 		Alle Business-Tabellen HIGH T1565

// Kritische Tabellen für EU1-Überwachung

Tabelle Inhalt Warum kritisch
USR02 Benutzer + Passwort-Hashes Credential Dump — alle SAP-Passwörter
USR04 Benutzerprofile Berechtigungsstruktur aller User
AGR_USERS Rollenzuordnungen Wer hat welche Rolle — für Privilege Mapping
RFCDES RFC-Verbindungen Verbundene Systeme — Lateral Movement Planung
T000 Client-Liste Mandantenstruktur — Recon für weitere Angriffe
ABAP_SOURCE ABAP-Quellcode Business Logic, Backdoors im Code verstecken
SPL Datenexfiltration via EU1 — massiver Tabellenzugriff in kurzer Zeit
index=sap_audit msgid="EU1" | where match(object, "(?i)(USR02|USR04|AGR_USERS|RFCDES|T000|ABAP_SOURCE)") | bucket _time span=60s | stats count as table_reads, sum(record_count) as total_records, values(object) as tables_accessed by _time, user, terminal | where total_records > 1000 | sort - total_records

Kill Chain — Event IDs pro Szenario

Im SAP-Umfeld laufen Angriffe oft in einer klaren Kette ab. Erst Recon, dann Zugangsverschaffung, dann Privilege Escalation und schließlich Datenexfiltration.

// SZENARIO 1 — SAP Brute-Force
AU2 × N AUI
Viele fehlgeschlagene Logins (AU2) von einer IP/Terminal → Konto automatisch gesperrt (AUI). Gleiche Zeitspanne, gleicher User oder Durchlauf mehrerer Usernamen = Password Spray.
// SZENARIO 2 — Kompromittiertes Admin-Konto
AU1 (02:30 Uhr) AU6 (SE38) AU6 (SU01) AU6 (PFCG)
Login nachts (AU1) + sofortiger Zugriff auf hochprivilegierte T-Codes wie SE38, SU01, PFCG (AU6). Keine normale Admin-Arbeit — typische Kill Chain eines kompromittierten Kontos.
// SZENARIO 3 — Privilege Escalation (AUB Storm)
AUF × 15 AUB
Benutzer TKURT versucht systematisch Zugriff auf PFCG, SE11, SM30, SM59, SU01 → 15x AUF (Authorization Failure). Dann AUB (Benutzer geändert): jemand hat ihm die Rollen gegeben.
// SZENARIO 4 — Externer RFC-Angriff
AU4 (externe IP) AU9 (RFC_READ_TABLE) EU1
Externe IP versucht RFC-Login mit Default-Usernamen (AU4) → bei Erfolg: RFC_READ_TABLE aufgerufen (AU9) → direkte Tabellenzugriffe (EU1) ohne GUI-Login.
// SZENARIO 5 — Datenexfiltration
EU1 (USR02) EU1 (AGR_USERS) EU1 (RFCDES) EU1 (ABAP_SOURCE)
Benutzer SDEMIR liest USR02, AGR_USERS, RFCDES und ABAP_SOURCE — über 20.000 Datensätze in einer Sekunde. Kein legitimer Benutzer braucht alle diese Tabellen in einem Durchgang.

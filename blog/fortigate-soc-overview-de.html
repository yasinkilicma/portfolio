Lab-Topologie

Das gesamte Lab läuft auf einem einzigen Mac mini M4 — FortiGate-VM nativ als ARM64 in UTM, ein Debian-Client hinter der Firewall über QEMU Socket-Networking, und Splunk als Docker-Container für die SOC-Pipeline. Kein separater Server nötig.

Internet (WAN-Simulation) │ ▼ FortiGate-VM 7.4.12 (UTM · ARM64 · Mac mini M4) │ port1 = WAN (192.168.64.x / UTM shared) │ port2 = LAN (10.0.0.1/24) │ ├── Debian Client (QEMU socket netdev · 10.0.0.2) │ └── Traffic wird vollständig durch die FGT inspiziert │ └── Splunk (Docker · Port 8000 Web · 8088 HEC) └── Python Syslog-to-HEC Bridge empfängt FortiGate-Logs
→

Apple Silicon Challenge: Die meisten FortiGate-Lab-Guides setzen x86 voraus. FortiOS 7.4.12 läuft jedoch auch als ARM64-Image — der entscheidende Trick ist die serielle Konsole über TCP in UTM, da kein normales Display verfügbar ist. Details im UTM-Setup Writeup.

Lab-Journey — Schritt für Schritt

01
// Lab Setup — Virtualisierung
FortiGate-VM nativ auf Apple Silicon mit UTM
UTM ARM64 Serielle Konsole TCP Mac mini M4
FortiGate-VM 7.4.12 (ARM64) in UTM auf einem Mac mini M4 — inklusive der Stolpersteine, die kein Tutorial erwähnt: serielle Konsole über TCP, initiales Interface-Mapping, Lizenz-Bypass für unlizenzierte FortiOS-VMs.
02
// Lab Setup — Networking
Einen Client hinter die FortiGate hängen — QEMU Socket-Networking
QEMU socket netdev L2 Bridge Debian Cloud Image
Einen Debian-Client so hinter die Firewall bringen, dass sein gesamter Traffic durch FortiGate inspiziert wird — über QEMU socket netdev als echtes isoliertes L2-Segment. Kein NAT, kein Hairpinning — echter Firewall-Betrieb.
03
// Detection — IPS Custom Signature
IPS testen mit einer Custom F-SBID-Signatur
F-SBID IPS 200 → DROP
Eine eigene F-SBID-Signatur schreibt, die ein bestimmtes Muster im HTTP-Traffic erkennt und blockt. Kontroll-Request: HTTP 200. Angriffs-Request mit dem Muster: dropped. Inklusive IPS-Log-Beweis.
04
// Detection — Antivirus
AntiVirus testen mit EICAR
EICAR HTTP Scan Connection Reset
Den AV-Scan der FortiGate deterministisch beweisen: saubere Datei kommt durch (HTTP 200), EICAR-Testdatei wird per Connection-Reset geblockt. AV-Log dokumentiert den Fund mit Signatur und Aktion.
05
// Detection — Network Forensics
Port-Scan-Erkennung mit FortiGate & Splunk
nmap SYN-Scan TCP-RST Fingerprint sentbyte=164
nmap SYN-Scan-Erkennung ohne IPS-Modul — nur anhand des TCP-RST-Fingerabdrucks in FortiGate Traffic-Logs: sentbyte=164 / rcvdbyte=60. Echtzeit-Dashboard in Splunk zeigt Scan-Muster sofort.
06
// SOC Pipeline — Live Dashboard
FortiGate-Logs live in Splunk — Echtzeit-SOC-Dashboard
Python HEC Bridge Splunk Dashboard 999 Events 34 blocked
FortiGate-Syslog per Python-HEC-Bridge live in Splunk einspeisen, ein sechsreihiges SOC-Dashboard bauen und mit EICAR + nmap echten Angriffsverkehr erzeugen: 999 Events, 34 geblockt, 9 Virus-Funde live im Dashboard.
07
// Investigation — Traffic Forensics
FortiGate Firewall Traffic Analyse — Network Forensics
C2 192.168.250.20 DNS Exfiltration 55.279 Events
Forensische Analyse auf 55.279 FortiGate-Events: C2-Kommunikation von 192.168.250.20 → 8.8.8.8 über ungewöhnliche Ports, externe Angreifer-IPs und DNS-Exfiltrations-Muster anhand überlanger Hostnamen und hoher Query-Frequenz.
08
// Assessment — Where the Firewall Failed
FortiGate Security Assessment — Wo die Firewall versagte
20% Effektivität C2-DNS-Bypass SMB Blind-Spot
Kritische Analyse der Firewall-Effektivität: C2-DNS-Bypass (DNS-over-HTTPS ungeblockt), SMB-Blind-Spot (East-West-Traffic nicht inspiziert), fehlende Segmentierung — nur 20% der Angriffe wurden geblockt.
09
// NSE4 Certification Prep
FortiGate NSE4: Hands-On vom ersten CLI-Befehl bis zum VPN-Tunnel
NSE4 FortiOS CLI 8 Übungen VPN
8 praktische Übungen für die NSE4-Prüfungsvorbereitung: CLI-Grundlagen, Firewall-Policies, NAT, Routing, Security Profiles, Debug-Befehle und VPN-Tunnel — mit echten Lab-Ausgaben aus dem FortiGate-VM-Lab.

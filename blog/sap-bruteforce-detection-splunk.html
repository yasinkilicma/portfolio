Rather than setting up a real SAP NetWeaver system (which would require a full ERP stack), I wrote a Python script that generates authentic SAP Security Audit Log events in the correct format and ships them directly to Splunk via HEC. This is the same approach used in red team simulations and SIEM validation exercises.

The SAP Security Audit Log

The SAP Security Audit Log (SAL) — accessed via transaction SM20 — is the primary data source for SAP threat detection. It records every security-relevant event in the system: logins, failed authentications, transaction starts, authorization failures, and RFC calls.

ℹ️

Why SAP logs matter for SOC: SAP ERP systems store the most sensitive business data — financial records, HR data, customer information. An attacker who gains SAP access can extract everything without touching any endpoint, bypassing traditional EDR and network-based detection entirely.

The key event IDs monitored in this lab:

AU2
Failed Dialog Login

Wrong password or locked user attempt. The primary brute-force indicator.

AU1
Successful Dialog Login

Successful logon. After a storm of AU2 events, a successful AU1 means credentials were cracked.

AUI
User Locked

Account automatically locked after exceeding the failed login threshold.

AUB
Authorization Check Failed

User tried to access a resource or transaction without sufficient permissions.

AU6
Transaction Started

A SAP transaction (tcode) was executed. Critical tcodes like SE38, SU01, PFCG are high-risk.

AU4
Failed RFC Login

Failed remote function call authentication — common in external SAP attack tooling.

EU1
Generic Table Read

Direct database table access. Reading USR02 or AGR_USERS is a major red flag.

DU9
Report / Program Started

ABAP report executed. User enumeration reports like RSUSR002 are suspicious after-hours.

The Attack — Brute-Force Scenario

The simulated attacker targets the SAP production system SAPPRD01, client 100. They identify a valid username — JSCHMIDT — through OSINT or a previous reconnaissance phase, and begin hammering the login interface from an external IP address.

🚨

Attacker IP: 45.33.32.156 — a known scanning IP address originating outside the corporate network. No VPN, no internal IP — this is a direct internet-facing SAP system attack.

Attack Timeline

09:00 — Attack Begins

Attacker starts sending login requests to SAPPRD01 targeting user JSCHMIDT. First AU2 event generated.

09:00–09:12 — 12 Failed Logins (AU2 × 12)

Twelve failed login attempts in rapid succession from 45.33.32.156. Each generates an AU2 event in the Security Audit Log.

09:13 — Account Locked (AUI)

SAP's automatic lockout policy triggers. User JSCHMIDT is locked. Severity: HIGH. The attacker is now blocked — but the account is also unavailable to the legitimate user.

09:45 — Legitimate User Returns

Real JSCHMIDT logs in from internal IP 10.10.1.42 after the account is unlocked by the basis team. Normal activity resumes.

Detection — Splunk SPL Queries

Query 1 — Identify the Brute-Force Source

The first query finds all users who received more than 5 failed login attempts (AU2) and groups them by source IP. A single IP generating multiple failures against one user is the classic brute-force pattern.

SPL Brute-force detection — failed logins per user/IP
# Find IPs with more than 5 failed logins against the same account index=sap_audit event_id=AU2 | stats count by user, terminal | where count > 5

→ Result: user JSCHMIDT targeted 12 times from external IP 45.33.32.156

The result is unambiguous: one user, one attacking IP, twelve attempts. In a production environment this would immediately trigger a P1 alert and an IP block at the firewall level.

Query 2 — Confirm the Account Lockout

A brute-force that stays below the lockout threshold is far more dangerous than one that gets caught. The AUI event confirms the lockout actually fired — and reveals exactly when it happened, which server processed it, and the severity classification.

SPL Account lockout confirmation — AUI event details
# Show full details of the account lockout event index=sap_audit event_id=AUI | table _time, user, terminal, server, severity, message

→ Result: JSCHMIDT locked on SAPPRD01 — severity HIGH — "User JSCHMIDT locked after 12 failed login attempts"

The AUI event is the closing proof. We now have the complete picture: attacker IP → victim account → lockout event → affected production server. This is everything needed for an incident ticket.

Key Findings

🔴
External IP Targeting Production SAP

45.33.32.156 is a known scanning IP with no business relationship to the organization. Direct internet access to SAP GUI port is a critical misconfiguration — SAP should never be internet-facing without a WAF or SAP Web Dispatcher.

🔒
Account Lockout as a DoS Vector

While the lockout policy stopped the brute-force, it also made the legitimate user JSCHMIDT unable to work. An attacker can weaponize this to lock out key SAP users (finance, HR) during critical business hours.

⚠️
Username Enumeration Risk

The attacker knew JSCHMIDT was a valid username — SAP's default error message doesn't differentiate between "wrong user" and "wrong password," but error timing differences can be exploited for enumeration.

✅
Lockout Policy Worked — But Alerting Was Missing

The account lockout policy correctly fired after 12 attempts. However, without SIEM integration, this event would have lived silently in the SM20 log — unnoticed until someone manually checked.

Recommendations

01
Never Expose SAP Directly to the Internet

Place a SAP Web Dispatcher or WAF in front of all SAP systems. Direct internet access to the SAP Router or DIAG port is a critical exposure.

02
Alert on AU2 Bursts in Real-Time

Configure SIEM alerts: >5 AU2 events from the same IP within 10 minutes = immediate P1. Don't wait for AUI — the lockout is already too late.

03
Lower the Lockout Threshold

SAP default is often 5–12 attempts. Reduce to 3 for privileged users (SAPABAP1, DDIC, SAP*). High-value accounts need tighter protection.

04
Block External IPs at the Firewall

Feed AU2 source IPs into your firewall automation. An IP that generates SAP login failures should be blocked within minutes, not hours.

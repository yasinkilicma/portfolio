Warum ein eigener Generator?

Für SOC-Übungen mit Splunk gibt es öffentliche Datasets wie das BOTS v1 (Cerber Ransomware). Für SAP-Logs existiert nichts Vergleichbares — SAP-Systeme laufen in abgeschotteten Unternehmensumgebungen, echte Logs sind nie öffentlich.

Die Lösung: ich habe das Format des SAP Security Audit Log (Transaktion SM20) analysiert und ein Python-Skript geschrieben, das realistische Events mit denselben Feldern und Event-IDs produziert wie ein echtes SAP-System.

→

Ziel: Splunk-Dashboard mit SAP-Daten füllen, ohne Zugang zu einem echten SAP-System zu benötigen — reproduzierbar, kontrolliert, anpassbar.

1 — Imports & Konstanten

Das Skript braucht nur zwei externe Pakete: requests für die HTTP-Kommunikation mit Splunk HEC und urllib3 um SSL-Warnungen zu unterdrücken (Self-signed Cert im Lab).

2 — Hilfsfunktionen

Zwei kleine Helfer die in jedem Szenario verwendet werden: ts() berechnet Zeitstempel und send() sendet Events an den HEC.

PYTHON Zeilen 65–102 — ts() und send()
def ts(base: datetime, offset_minutes: int = 0) -> str: return (base + timedelta(minutes=offset_minutes)).strftime("%Y-%m-%dT%H:%M:%S") ↑ base + offset → "2026-06-16T09:08:00" (ISO 8601 Format wie SAP es schreibt) ↑ Jedes Szenario ruft ts(base, i) auf → Events sind zeitlich geordnet def send(events: list[dict], token: str): headers = { "Authorization": f"Splunk {token}", ↑ HEC-Authentifizierung: "Splunk " + Token (kein Bearer!) "Content-Type": "application/json" } ok = fail = 0 for e in events: payload = { "index": "sap_audit", # Ziel-Index "sourcetype": "sap:audit", # Parser-Hinweis für Splunk "event": e # eigentliches Event-Dict } r = requests.post( SPLUNK_HEC_URL, headers=headers, data=json.dumps(payload), verify=False, # Self-signed SSL ignorieren timeout=5 ) if r.status_code == 200: ok += 1 else: fail += 1 ↑ Jedes Event wird einzeln gesendet → bei Fehler laufen die anderen weiter

PYTHON Zeilen 107–153 — scenario_brute_force()
def scenario_brute_force(base: datetime) -> list[dict]: evts = [] victim = "JSCHMIDT" attacker_ip = random.choice(IPS["external"]) ↑ Zufällige externe IP bei jedem Aufruf → unterschiedliche Logs möglich for i in range(12): evts.append({ "timestamp": ts(base, i), # t+0min, t+1min, … t+11min "event_id": "AU2", "user": victim, "terminal": attacker_ip, "message": f"Login failed … (attempt {i+1})", }) ↑ 12 fehlgeschlagene Logins = AU2-Storm → typisches Brute-Force-Muster in SM20 evts.append({ "timestamp": ts(base, 13), "event_id": "AUI", # Account gesperrt "severity": "HIGH", "message": f"User {victim} locked after 12 failed login attempts", }) ↑ AUI folgt immer nach einer konfigurierbaren Anzahl AU2-Events (SAP Parameter login/fails_to_user_lock) evts.append({ "timestamp": ts(base, 45), # 45 Minuten später "event_id": "AU1", # echter Login vom internen Netz "terminal": random.choice(IPS["internal"]), }) ↑ Zeigt dass JSCHMIDT danach entsperrt wurde und sich normal einloggt return evts

def scenario_data_exfiltration(base: datetime) -> list[dict]: evts = [] user = "SDEMIR" ip = random.choice(IPS["internal"]) ↑ Interne IP → Insider Threat oder kompromittiertes internes Konto for i, table in enumerate(SENSITIVE_TABLES): evts.append({ "timestamp": ts(base, i * 3 + 180), ↑ +180 Minuten = 3 Stunden nach base (klar von anderen Szenarien getrennt) "event_id": "EU1", "table": table, "rows_read": random.randint(500, 5000), ↑ Zufällige Zeilenzahl → realistischer als immer dieselbe Zahl "severity": "HIGH" if table in ["USR02", "USR04"] else "MEDIUM", ↑ USR02/USR04 automatisch HIGH — enthält Passwort-Hashes }) return evts

Erkenntnisse & Hinweise

⚠

HTTPS Pflicht: Splunk HEC erfordert HTTPS (Port 8088 mit SSL). HTTP-Requests werden mit RemoteDisconnected abgelehnt. Im Lab: verify=False in requests.post() zum Ignorieren des Self-signed Zertifikats.

→

Field-Namen prüfen: Splunk extrahiert Fields aus dem JSON automatisch. Das Feld terminal landet als eigenes Splunk-Field — aber nicht als src_ip. Immer mit | head 1 | fields * prüfen welche Fields wirklich vorhanden sind.

✓

Erweiterbar: Neue Szenarien lassen sich einfach als eigene Funktion hinzufügen und in main() einbinden. Die Event-ID-Tabelle SAP_EVENTS deckt bereits alle 16 SAP Security Audit Log Event-Typen ab.

with SNMP & LLDP

A Python tool that automatically discovers network devices via SNMP, reads LLDP and CDP neighbour tables, builds a topology graph using NetworkX, and visualises it as a hierarchical PNG or an interactive D3.js HTML map.

The Problem

Network documentation is almost always out of date. Physical changes happen, diagrams don't get updated, and a few months later nobody knows what connects to what. This tool solves that by auto-discovering the topology directly from the devices using protocols they're already broadcasting.

How SNMP Discovery Works

SNMP (Simple Network Management Protocol) organises all device information in a tree structure called the MIB (Management Information Base). Every data point has a unique address called an OID. The tool queries three categories of OIDs:

📋
System OIDs (Standard MIB-2)

sysName (1.3.6.1.2.1.1.5.0), sysDescr (1.3.6.1.2.1.1.1.0), sysLocation — supported by virtually every SNMP-capable device regardless of vendor.

🔗
LLDP MIB (IEEE 802.1AB)

lldpRemSysName, lldpRemPortId — vendor-neutral neighbour discovery. Supported by Cisco, HP, Juniper, Aruba, and Linux (lldpd).

🔶
CDP MIB (Cisco proprietary)

cdpCacheDeviceId, cdpCacheDevicePort — Cisco Discovery Protocol. Provides neighbour device name, port, and platform on Cisco IOS devices.

Hierarchical Layout vs. Spring Layout

The first version used NetworkX's default spring_layout (Fruchterman-Reingold algorithm), which places nodes based on simulated physical forces. The result was aesthetically random — core switches ended up in corners, workstations in the middle.

💡

Fix: I replaced spring_layout with a custom hierarchical layout that assigns nodes to layers based on their role: Router/Firewall at the top, Core Switches below, Access Switches next, then Servers, then Workstations — matching the actual network architecture.

Layer 0 ●─────────● Router / Firewall │ │ Layer 1 ●────● ●────● Core Switches │ │ Layer 2 ● ● ● ● Access Switches │ │ │ Layer 3 ● ● ● Servers │ Layer 4 ● ● ● ● ● Workstations

Node sizes also scale with the connection degree — a Core Switch with 4 connections appears larger than a Workstation with 1, instantly communicating device importance.

Interactive HTML Output (D3.js)

The HTML output uses a D3.js force-directed simulation. Device data is embedded as JSON directly in the HTML file — no server needed, just open in a browser. Features include:

🖱️
Drag & Drop

Nodes can be repositioned manually by dragging, with the physics simulation resuming on release.

🔍
Zoom & Pan

Scroll to zoom, drag the background to pan. Useful for large topologies with many devices.

💬
Hover Tooltips

Hovering a node shows device name, IP address, type, location, and sysDescr excerpt.

Demo Mode

The tool includes a --demo flag that loads a pre-built 14-device topology from a YAML file — no SNMP-capable hardware required. The demo includes a router, firewall, two core switches, two access switches, three servers, and five workstations.

BASHRun without SNMP hardware
python mapper.py --demo --ausgabe html # Opens a D3.js topology map in output/topologie_*.html python mapper.py --demo --ausgabe png # Generates hierarchical PNG in output/topologie_*.png # Live scan (requires SNMP access) python mapper.py --netzwerk 192.168.1.0/24 --community public

Key Takeaways

01
LLDP is Underused

Most managed switches broadcast LLDP by default. Querying it via SNMP gives you a complete neighbour map with zero active probing.

02
Layout Matters

A topology diagram is only useful if it matches the viewer's mental model. Hierarchical layers communicate network structure instantly.

03
SNMP = Free Intelligence

sysDescr alone identifies device type, OS version, and vendor — without active fingerprinting or vulnerability scanning.

04
Demo Mode Matters

A portfolio project that requires specific hardware to run is a portfolio project nobody can evaluate. Demo mode makes it testable anywhere.

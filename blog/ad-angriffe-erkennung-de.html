Warum Active Directory? AD ist das Herzstück jedes Windows-Unternehmensnetzwerks. 95% der Fortune-500-Unternehmen nutzen AD. Angreifer zielen immer auf AD ab — wer AD kompromittiert, hat das gesamte Netzwerk unter Kontrolle.

Kerberos — Kurz erklärt

Kerberos ist das Standard-Authentifizierungsprotokoll in Active Directory. Es basiert auf Tickets statt Passwörtern. Alle fünf Angriffe in diesem Artikel nutzen Schwachstellen im Kerberos-Protokoll aus.

// SCHRITT 1 — AS-REQ / AS-REP
Client → Domain Controller: "Ich bin Max, gib mir ein TGT"
DC prüft Pre-Authentication (verschlüsselt mit Passwort-Hash) → EventID 4768
// SCHRITT 2 — TGS-REQ / TGS-REP
Client → DC: "Ich habe ein TGT, gib mir ein Service Ticket für SQL-Server"
DC stellt Service Ticket aus (verschlüsselt mit Service Account Hash) → EventID 4769
// SCHRITT 3 — AP-REQ
Client → Service: "Hier ist mein Service Ticket"
Service entschlüsselt Ticket mit eigenem Hash → Zugriff gewährt

01 — Kerberoasting

Kerberoasting ist der häufigste AD-Angriff in der Praxis. Jeder Domänenbenutzer kann TGS-Tickets für Service Accounts anfordern. Das Ticket ist mit dem Service-Account-Hash verschlüsselt — und kann offline gecracked werden.

⚠

Warum funktioniert das? RC4-HMAC (EncryptionType 0x17) ist schwächer als AES256 (0x12). Wenn ein Angreifer ein RC4-verschlüsseltes TGS-Ticket anfordert, kann er es mit Hashcat in Stunden cracken. Service Accounts haben oft schwache Passwörter und keine regelmäßige Rotation.

// Erkennungsmuster

EventID 4769 mit TicketEncryptionType = 0x17 (RC4) für einen Service Account (SPN-Name). Mehrere TGS-Anfragen für verschiedene Service Accounts in kurzer Zeit = Scanning.

SPL Kerberoasting — RC4 TGS-Requests für Service Accounts
index=windows_security EventCode=4769 | where TicketEncryptionType="0x17" AND ServiceName!="krbtgt" AND NOT match(ServiceName, "^\$") ← Computer-Accounts ausschließen AND NOT match(ServiceName, "^[A-Z0-9-]+\$$") ← DC-Accounts ausschließen | bucket _time span=10m | stats count as tgs_requests, dc(ServiceName) as unique_services, values(ServiceName) as targeted_services by _time, AccountName, IpAddress | where unique_services >= 3 ← 3+ verschiedene SPNs = Scanning | sort - tgs_requests

02 — AS-REP Roasting

AS-REP Roasting zielt auf Konten ab, bei denen Kerberos Pre-Authentication deaktiviert ist ("Do not require Kerberos preauthentication"). Ohne Pre-Auth kann jeder anonym ein TGT anfordern — das verschlüsselt mit dem Passwort-Hash des Benutzers zurückkommt.

!

Unterschied zu Kerberoasting: Kerberoasting braucht einen Domain User Account. AS-REP Roasting funktioniert anonym — kein Passwort, kein Account nötig. Nur die Liste der anfälligen Usernames reicht.

// Erkennungsmuster

EventID 4768 mit PreAuthType = 0 (kein Pre-Auth) und EncryptionType = 0x17 (RC4). Legitime Logins haben PreAuthType = 2.

SPL AS-REP Roasting — TGT ohne Pre-Authentication
index=windows_security EventCode=4768 | where PreAuthType="0" ← kein Pre-Auth = anfällig AND TicketEncryptionType="0x17" ← RC4 = schwach AND Status="0x0" ← nur erfolgreiche Anfragen | stats count as asrep_requests, values(TargetUserName) as vulnerable_users by IpAddress | sort - asrep_requests
SPL Anfällige Konten identifizieren — Pre-Auth deaktiviert
// Alle User finden, die ohne Pre-Auth TGTs anfordern index=windows_security EventCode=4768 PreAuthType="0" | stats count by TargetUserName, IpAddress | sort TargetUserName // Diese User sollten sofort Pre-Auth aktiviert bekommen

03 — DCSync

DCSync nutzt das AD-Replikationsprotokoll aus. Normalerweise synchronisieren Domain Controller untereinander Passwort-Hashes. Mimikatz kann diese Replikation simulieren — und damit alle Hashes aus dem DC extrahieren, ohne sich physisch am DC anzumelden.

⚠

Voraussetzung für den Angreifer: Das kompromittierte Konto braucht DS-Replication-Get-Changes und DS-Replication-Get-Changes-All Rechte. Diese haben standardmäßig: Domain Admins, Enterprise Admins, Domain Controller — und leider manchmal auch fehlerhaft konfigurierte Service Accounts.

// Erkennungsmuster

EventID 4662 mit spezifischen Replikations-GUIDs, nicht von einem Domain Controller ausgeführt (kein Computerkonto). Das ist der Schlüssel: echte DC-Replikation kommt von DC-Computerkonten ($), DCSync kommt von User-Accounts.

SPL DCSync Erkennung — Replikation von Nicht-DC-Konten
index=windows_security EventCode=4662 AccessMask="0x100" | where ( match(Properties, "1131f6aa-9c07-11d1-f79f-00c04fc2dcd2") ← DS-Replication-Get-Changes OR match(Properties, "1131f6ad-9c07-11d1-f79f-00c04fc2dcd2") ← DS-Replication-Get-Changes-All ) | where NOT match(SubjectUserName, "\$$") ← Computer-Konten (DC$) ausschließen | table _time, SubjectUserName, SubjectDomainName, ComputerName | sort _time
!

Jedes 4662 von einem User-Account mit diesen GUIDs = sofortiger Alarm. Es gibt keinen legitimen Grund, warum ein normales Benutzerkonto Replikationsrechte ausübt. Das ist immer entweder ein Angriff oder eine schwerwiegende Fehlkonfiguration.

04 — Golden Ticket

Der gefährlichste AD-Angriff. Mit dem Hash des KRBTGT-Kontos (das Master-Key-Konto für alle Kerberos-Tickets in der Domain) kann ein Angreifer beliebige TGTs fälschen — für jeden Benutzer, mit jeder Gruppe, mit beliebiger Gültigkeit (bis zu 10 Jahre).

!

Warum Golden Ticket so gefährlich ist: Selbst nach einem Passwort-Reset des Administrator-Kontos bleibt der Angreifer drin — das Golden Ticket funktioniert weiter, bis der KRBTGT-Hash zweimal zurückgesetzt wird. Viele Unternehmen wissen das nicht.

// Erkennungsmuster

Golden Tickets sind schwer zu erkennen — sie sehen aus wie echte Tickets. Anomalien: (1) EventID 4624 ohne vorheriges 4768 vom gleichen Client, (2) ungewöhnliche Ticket-Laufzeiten (>10h), (3) SID-History enthält Domain-Admin-SID bei normalem User-Account.

SPL Golden Ticket Indikator — Login ohne TGT-Anfrage vom gleichen Host
// Schritt 1: Alle erfolgreichen Netzwerk-Logins index=windows_security EventCode=4624 LogonType=3 AuthenticationPackageName="Kerberos" | eval login_key=TargetUserName+"|"+IpAddress | table _time, login_key, TargetUserName, IpAddress, ComputerName // Schritt 2: TGT-Anfragen in gleicher Zeit index=windows_security EventCode=4768 Status="0x0" | eval tgt_key=TargetUserName+"|"+IpAddress | table _time, tgt_key // Schritt 3: 4624 ohne passendes 4768 = Golden Ticket Verdacht // (Korrelation im Splunk Enterprise Security Framework)
SPL Golden Ticket — Anomalie-Erkennung via Privilegien
index=windows_security (EventCode=4624 OR EventCode=4672) | eval suspicious_privs = if( match(PrivilegeList, "(?i)(SeTcbPrivilege|SeCreateTokenPrivilege)"), "GOLDEN-TICKET-INDIKATOR", "normal" ) | where suspicious_privs="GOLDEN-TICKET-INDIKATOR" | table _time, EventCode, TargetUserName, IpAddress, PrivilegeList | sort _time

05 — Pass-the-Ticket

Pass-the-Ticket (PtT) ist der Kerberos-Äquivalent zu Pass-the-Hash. Statt NTLM-Hashes werden gestohlene Kerberos-Tickets (TGT oder TGS) in eine andere Session injiziert. Mimikatz: sekurlsa::tickets /export → kerberos::ptt ticket.kirbi.

// Unterschied zu Pass-the-Hash

Merkmal Pass-the-Hash Pass-the-Ticket
Was wird gestohlen? NTLM-Hash (aus LSASS) Kerberos Ticket (.kirbi)
Protokoll NTLM Kerberos
Event ID (Login) 4776 + 4624 Type 3 NTLM 4768/4769 + 4624 Type 3 Kerberos
Gültigkeit Bis Hash geändert Bis Ticket abläuft (10h TGT / 8h TGS)
Erkennbar an NTLM bei Kerberos-fähigen Systemen Ticket-Nutzung ohne passende TGT-Anfrage
SPL Pass-the-Ticket — TGS-Nutzung ohne vorherige TGT-Anfrage
// TGS-Anfragen (4769) ohne passendes TGT (4768) in kurzer Zeit index=windows_security EventCode=4769 | eval tgs_time=_time | join type=left AccountName [ search index=windows_security EventCode=4768 Status="0x0" | eval tgt_time=_time | table AccountName, tgt_time ] | eval time_diff = abs(tgs_time - tgt_time) | where isnull(tgt_time) OR time_diff > 3600 ← kein TGT in letzter Stunde | table _time, AccountName, ServiceName, IpAddress | sort _time

Kill Chain — AD-Kompromittierung

In der Praxis werden diese Angriffe kombiniert. Eine typische AD-Kompromittierung läuft in dieser Reihenfolge ab:

// PHASE 1 — Erstzugang & Reconnaissance
4625 / 4771 4624 Type 3
Phishing oder Brute-Force verschafft initialen Zugang zu einem Domain User Account. Fehlgeschlagene Logins häufen sich (4625/4771).
// PHASE 2 — Privilege Escalation via Kerberoasting
4769 (RC4 × N)
Angreifer fordert TGS-Tickets für mehrere Service Accounts mit RC4-Verschlüsselung an (4769). Offline-Cracking → Service-Account-Passwort erlangt.
// PHASE 3 — Domain Compromise via DCSync
4662 (Replikations-GUIDs) 4672
Mit Domänenadmin-Rechten: DCSync via Mimikatz → alle NTLM-Hashes inklusive KRBTGT extrahiert (4662 von User-Account). Admin-Login mit SeDebugPrivilege (4672).
// PHASE 4 — Persistence via Golden Ticket
4624 (kein 4768) 4672 4728
KRBTGT-Hash → Golden Ticket erstellt → Angreifer kann sich als beliebiger User ausgeben (4624 ohne 4768). Zusätzlich neues Backdoor-Konto in Domain Admins (4728) für dauerhafte Kontrolle.

Master Query — AD Kill Chain Detection

SPL Vollständige AD Kill Chain — alle kritischen Events in einem Query
index=windows_security ( EventCode=4769 TicketEncryptionType="0x17" ← Kerberoasting OR (EventCode=4768 PreAuthType="0") ← AS-REP Roasting OR EventCode=4662 ← DCSync OR EventCode=4771 ← Kerberos Brute-Force OR (EventCode=4624 LogonType=3 AuthenticationPackageName="Kerberos") OR EventCode=4672 OR EventCode=4728) ← Domain Admin Membership | eval attack_phase = case( EventCode=4769 AND TicketEncryptionType="0x17", "Kerberoasting", EventCode=4768 AND PreAuthType="0", "AS-REP-Roasting", EventCode=4662, "DCSync", EventCode=4771, "Kerberos-BruteForce", EventCode=4728, "Domain-Admin-Persistence", EventCode=4672, "Admin-Privileges", 1=1, "Lateral-Movement" ) | eval severity = case( EventCode=4662, "CRITICAL", ← DCSync immer kritisch EventCode=4728, "CRITICAL", EventCode=4769 AND TicketEncryptionType="0x17", "HIGH", EventCode=4768 AND PreAuthType="0", "HIGH", 1=1, "MEDIUM" ) | table _time, severity, attack_phase, EventCode, TargetUserName, ServiceName, IpAddress, ComputerName | sort _time

Warum Event IDs? Windows protokolliert Sicherheitsereignisse im Security Event Log (eventvwr.msc → Windows Logs → Security). Jedes Ereignis hat eine eindeutige ID. SOC-Analysten filtern gezielt nach diesen IDs, um Angriffe zu erkennen — manuell oder per SIEM.

01 — Authentication & Logon

Anmeldeereignisse sind die Grundlage jeder SOC-Analyse. Sie zeigen, wer sich wann und von wo anmeldet — und ob Angriffe wie Brute-Force oder Pass-the-Hash stattfinden.

Event ID Beschreibung Angriff Priorität MITRE
4624 Erfolgreicher Login
Logon Type 2=Interaktiv · 3=Netzwerk · 10=Remote · 7=Unlock 		Lateral Movement, Pass-the-Hash (Type 3) MEDIUM T1078
4625 Fehlgeschlagener Login
Status 0xC000006A=falsch Passwort · 0xC0000064=User existiert nicht 		Brute-Force, Password Spray CRITICAL T1110
4634 Abmeldung
Session-Ende, Logon-ID korrelierbar mit 4624 		Session-Tracking, ungewöhnlich kurze Sessions INFO —
4647 Benutzer-initiierte Abmeldung
Explizites Logout (statt Timeout) 		Session-Forensik INFO —
4648 Login mit expliziten Credentials
runas, WMI, PsExec — andere Anmeldedaten als aktuelle Session 		Pass-the-Hash, Lateral Movement CRITICAL T1550.002
4672 Special Privileges zugewiesen
SeDebugPrivilege, SeImpersonatePrivilege bei Login 		Privilege Escalation, Admin-Zugang HIGH T1134
4740 Konto gesperrt (Lockout)
Nach x fehlgeschlagenen Versuchen — Quelle: Caller Computer Name 		Brute-Force, automatischer Alarm CRITICAL T1110
4776 NTLM-Authentifizierung (Domain Controller)
Error Code 0xC000006A=falsch · 0x0=Erfolg 		Pass-the-Hash, NTLM-Relay CRITICAL T1550.002
SPL Brute-Force Detection — 10+ Fehlversuche in 5 Minuten
index=windows_security EventCode=4625 | bucket _time span=5m | stats count by _time, TargetUserName, IpAddress | where count > 10 | sort - count
SPL Pass-the-Hash Erkennung — NTLM + Explicit Credentials Korrelation
index=windows_security (EventCode=4776 OR EventCode=4648) | eval attack_type = if(EventCode=4776, "NTLM-Auth", "Explicit-Creds") | table _time, attack_type, TargetUserName, WorkstationName, IpAddress | sort _time

index=windows_security (EventCode=4720 OR EventCode=4728 OR EventCode=4732) | eval event_type = case( EventCode=4720, "Neues Konto", EventCode=4728, "Domain Admin +Mitglied", EventCode=4732, "Local Admin +Mitglied" ) | table _time, event_type, TargetUserName, SubjectUserName, ComputerName | sort _time

03 — Process Execution

Prozess-Events zeigen, welche Programme gestartet wurden. Besonders wertvoll: die Parent-Child-Beziehung. Ein Word-Dokument, das cmd.exe startet, ist sofort verdächtig.

⚠

Voraussetzung: Process Creation Auditing muss aktiviert sein: gpedit.msc → Computer Configuration → Windows Settings → Security Settings → Advanced Audit Policy → Detailed Tracking → Audit Process Creation → Success

Event ID Beschreibung Angriff Priorität MITRE
4688 Neuer Prozess gestartet
NewProcessName, ParentProcessName, CommandLine — vollständige Prozesskette 		Malware Execution, Living-off-the-Land CRITICAL T1059
4689 Prozess beendet
Laufzeit berechenbar: 4689._time - 4688._time 		Short-lived Process Forensik INFO —
4104 PowerShell Script Block Logging
Vollständiger PowerShell-Code im Log — auch bei obfuszierten Skripten 		PowerShell-basierte Angriffe, Encoded Commands CRITICAL T1059.001
SPL Macro Malware — verdächtige Parent-Child Prozessketten
index=windows_security EventCode=4688 | where match(ParentProcessName, "(?i)(winword|excel|powerpnt|outlook)") AND match(NewProcessName, "(?i)(cmd|powershell|wscript|cscript|mshta)") | table _time, ParentProcessName, NewProcessName, CommandLine, ComputerName | sort _time
SPL Living-off-the-Land — verdächtige System-Tools als Angreifer-Werkzeug
index=windows_security EventCode=4688 | where match(NewProcessName, "(?i)(net\.exe|net1\.exe|whoami|ipconfig|nltest|mimikatz)") | table _time, NewProcessName, CommandLine, SubjectUserName, ComputerName | sort _time

04 — Persistence

Nach der Kompromittierung sichert der Angreifer seinen Zugang. Scheduled Tasks und Services überleben einen Neustart — und fallen oft wochenlang nicht auf.

Event ID Beschreibung Angriff Priorität MITRE
4698 Scheduled Task erstellt
TaskName, TaskContent (XML) — oft als "Windows Update" getarnt 		Persistence via Scheduled Task CRITICAL T1053.005
4702 Scheduled Task geändert
Bestehender Task modifiziert — subtiler als Neuerstellung 		Persistence, stealthy Backdoor HIGH T1053.005
7045 Neuer Service installiert
ServiceName, ImagePath — oft aus C:\Windows\Temp\ oder %APPDATA% 		Malicious Service Installation CRITICAL T1543.003
7036 Service Status geändert
started / stopped — Angreifer startet seinen Service 		Malicious Service Activation MEDIUM T1543.003
SPL Persistence Erkennung — verdächtige Tasks und Services
index=windows_security (EventCode=4698 OR EventCode=7045) | eval persist_type = if(EventCode=4698, "Scheduled Task", "Neuer Service") | eval suspicious = if( match(TaskContent, "(?i)(EncodedCommand|FromBase64|WebClient|DownloadString)") OR match(ImagePath, "(?i)(temp|appdata|programdata)"), "JA", "nein" ) | table _time, persist_type, suspicious, TaskName, SubjectUserName, ComputerName | sort _time

05 — Audit & Policy Manipulation

Das gefährlichste Signal: der Angreifer löscht Logs oder deaktiviert Auditing. Er weiß, dass er beobachtet wird — und versucht, Beweise zu vernichten.

Event ID Beschreibung Angriff Priorität MITRE
1102 Security Log gelöscht
Wer hat den Log geleert? SubjectUserName — immer sofort alarmieren 		Cover Tracks — IMMER kritisch CRITICAL T1070.001
4719 Audit Policy geändert
Logging deaktiviert — Angreifer will unsichtbar werden 		Defense Evasion CRITICAL T1562.002
4946 Windows Firewall Regel hinzugefügt
Neue Allow-Regel — Backdoor-Port öffnen 		Firewall Manipulation, C2 Channel HIGH T1562.004
4964 Special Groups Login
Privilegiertes Konto meldet sich an — Monitoring-Trigger 		Privileged Account Usage Tracking HIGH T1078
!

EventID 1102 = sofortiger Alarm. Kein legitimer Admin löscht den Security Log ohne Ticket. Wenn dieser Event erscheint: Incident Response sofort einleiten, da vorherige Spuren vernichtet wurden.

Angriffs-Korrelation — Event IDs pro Szenario

Einzelne Event IDs sind selten ausreichend. Erst die Kombination mehrerer IDs in kurzer Zeit zeigt einen echten Angriff.

// SZENARIO 1 — Brute-Force Attack
4625 × 10+ 4740
Viele fehlgeschlagene Logins (4625) in kurzer Zeit → Konto gesperrt (4740). Quelle: eine IP oder ein Host.
// SZENARIO 2 — Pass-the-Hash & Lateral Movement
4776 4648 4624 Type 3 4672
NTLM-Auth ohne Passwort (4776) + Explicit Credentials (4648) + Netzwerk-Login (4624 Typ 3) + Admin-Rechte (4672).
// SZENARIO 3 — Privilege Escalation
4720 4732 4728
Neues Konto (4720) → lokale Admin-Gruppe (4732) → Domain Admins (4728) — alles in wenigen Minuten.
// SZENARIO 4 — Persistence
4698 7045 7036
Scheduled Task (4698) oder neuer Service (7045) → Service startet (7036). ImagePath aus Temp-Verzeichnis = sofortiger Alarm.
// SZENARIO 5 — Macro Malware
4688 × 4
WINWORD.exe → cmd.exe → powershell.exe → net.exe. Alles 4688, aber Parent-Child-Kette macht den Angriff sichtbar.
// SZENARIO 6 — Cover Tracks (Gefährlichstes Signal)
1102 4719
Log gelöscht (1102) oder Auditing deaktiviert (4719) → Angreifer weiß, dass er beobachtet wird. Sofort IR einleiten.

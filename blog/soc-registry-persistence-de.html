🔴 PERSISTENCE REGISTRY RUN KEYS

Registry Persistence Analyse

Wie Cerber Ransomware mit Windows-Registry-Run-Keys für Auto-Start sorgt: Eine Persistence-Analyse auf 74.720 Registry-Events aus dem BOTS v1 Datensatz.

Registry Run Keys Persistence Cerber Ransomware Splunk SPL
74.720 Registry Events
24 Run-Key Events
4 Unique Run Keys

Kontext

In der Sysmon-Analyse haben wir osk.exe als Masquerading-Malware identifiziert. Die Frage: Wie stellt die Malware sicher, dass sie bei jedem Systemstart automatisch läuft?

Windows Registry Persistence

Die Windows-Registry enthält spezielle Run-Keys, die Programme bei jedem Login oder Systemstart automatisch ausführen. Malware nutzt diese Keys, um Persistence (Beständigkeit) zu erreichen.

Die wichtigsten Run-Keys:

  • HKLM\Software\Microsoft\Windows\CurrentVersion\Run → Für alle Benutzer (Systemweit)
  • HKCU\Software\Microsoft\Windows\CurrentVersion\Run → Nur für aktuellen Benutzer
  • HKCU\Software\Microsoft\Windows\CurrentVersion\RunOnce → Einmalig beim nächsten Login
  • HKLM\Software\Microsoft\Command Processor\AutoRun → Bei jedem CMD-Start
ℹ️

Warum Run-Keys? Einmal in die Registry geschrieben, läuft die Malware bei jedem Systemstart automatisch — ohne dass der Benutzer etwas bemerkt. Das ist die am häufigsten verwendete Persistence-Methode für Windows-Malware.

Die Analyse

Wir filtern alle Registry-Events, die Run-Keys betreffen, und gruppieren nach Key-Pfad und Wert.

SPL Registry-Run-Keys filtern und aggregieren
index=botsv1 earliest=0 sourcetype=WinRegistry # Nur Run-Keys (Auto-Start) | search key_path="*Run*" # Nach Key-Pfad und Daten aggregieren | stats count by key_path, data | sort -count | head 20

Das Ergebnis zeigt 14 unique Run-Key-Einträge aus 24 total Events. Die Top-Einträge sind eindeutig schädlich.

Abb. 1 — Registry-Run-Keys: osk.exe in Run und RunOnce, internat.exe als System-Weit-Persistence

Die Erkenntnisse

🔴 Kritisch: osk.exe in Run-Keys

Was wir sehen:

  • HKCU\...\Run\osk (2 Events) → osk.exe wird bei jedem Login gestartet
  • HKCU\...\RunOnce\osk (2 Events) → osk.exe läuft einmalig beim nächsten Login

Das ist die Persistence-Methode der Cerber-Ransomware! Die Malware nutzt den gleichen Namen osk.exe (On-Screen Keyboard), den wir bereits in der Sysmon-Analyse als Masquerading identifiziert haben. Jetzt sehen wir auch, wie sie persistiert.

🚨

Persistence erkannt: osk.exe ist in die Registry-Run-Keys eingetragen. Das bedeutet: Selbst wenn der Benutzer die Malware-Prozesse killt, wird sie beim nächsten Login automatisch neu gestartet. Die einzige Lösung ist die Registry-Bereinigung oder eine Isolation des Hosts.

🟡 internat.exe — System-Weite Persistence

HKCU\...\Run\internat.exe (5 Events) ist ein weiterer Indikator. internat.exe (International Settings) ist ein legitimer Windows-Prozess, aber in diesem Kontext — als Registry-Eintrag mit hoher Event-Anzahl — ist er verdächtig.

Warum ist das verdächtig? Ein legitimer internat.exe würde nicht manuell in die Run-Keys eingetragen werden. Der Registry-Eintrag deutet darauf hin, dass die Malware einen weiteren legitimen Prozessnamen für Persistence missbraucht.

🟡 Command Processor AutoRun

HKCU\...\Command Processor\AutoRun (2 Events) ist besonders gefährlich. Dieser Key führt automatisch einen Befehl aus, sobald der Benutzer eine CMD-Konsole öffnet.

Typischer AutoRun-Angriff:

CMD Beispiel eines schädlichen AutoRun-Befehls
# In HKCU\Software\Microsoft\Command Processor\AutoRun gespeichert: powershell.exe -WindowStyle Hidden -enc UwB0AGEAcgB0AC0AUwBsAGUAZQBwACAALQBzACAAMQAwADsAIABJAG4AdgBvAGsAZQAtAEUAeABwAHIAZQBzAHMAaQBvAG4A # Dieser Befehl wird ausgeführt, sobald CMD geöffnet wird!

Attack-Chain

Basierend auf den drei Analysen (Suricata, SMB, Sysmon, Registry) lässt sich die komplette Attack-Chain rekonstruieren:

  1. Reconnaissance: Nessus-Scan kartiert das Netzwerk (192.168.250.20 primäres Ziel)
  2. Initial Access: Exploit/SMB-Lateral-Movement bringt Malware auf 192.168.250.100
  3. Execution: osk.exe (Masquerading) wird ausgeführt
  4. Persistence: Registry-Run-Keys (Run, RunOnce) sorgen für Auto-Start
  5. Lateral Movement: SMB-Verbindung 192.168.250.100 → 192.168.250.20
  6. C2 Communication: Suricata detektiert .onion-Domain-Lookup (Cerber)
  7. Impact: File Encryption auf 192.168.250.20

Defensive Maßnahmen

IOC Registry-Indikatoren für Detection
# 1. Run-Keys mit ungewöhnlichen Prozessnamen key_path="*\Run\*" AND ( data="*osk.exe*" OR data="*internat.exe*" OR data="*AppData*" OR data="*Temp*" ) # 2. RunOnce-Keys (einmalige Ausführung) key_path="*RunOnce*" # 3. Command Processor AutoRun (CMD-Hijacking) key_path="*Command Processor*" AND key_path="*AutoRun*" # 4. Registry-Changes aus Benutzer-Verzeichnissen data="*Users\*" AND data="*.exe*"

SPL-Queries

SPL Komplette Registry-Persistence-Analyse
# 1. Alle Run-Keys mit Prozessnamen index=botsv1 earliest=0 sourcetype=WinRegistry | search key_path="*Run*" | stats count by key_path, data | sort -count # 2. Nur schädliche / verdächtige Einträge index=botsv1 earliest=0 sourcetype=WinRegistry | search key_path="*Run*" | search data="*osk.exe*" OR data="*internat.exe*" OR data="*AppData*" | table _time, key_path, data, process_image # 3. Command Processor AutoRun (CMD-Hijacking) index=botsv1 earliest=0 sourcetype=WinRegistry | search key_path="*Command Processor*" AND key_path="*AutoRun*" | table _time, key_path, data, process_image

Fazit

Diese Registry-Analyse vervollständigt das Bild der Cerber-Ransomware-Attacke:

  1. Suricata: Cerber-Ransomware auf 192.168.250.20 detektiert
  2. SMB: Lateral Movement von 192.168.250.100 → 192.168.250.20
  3. Sysmon: osk.exe Masquerading identifiziert
  4. Registry: osk.exe in Run-Keys für Persistence eingetragen

Die Registry ist der "Schlüssel" zur Malware-Persistence. Ohne diese Analyse hätten wir nicht verstanden, warum die Ransomware nach einem Reboot wieder erscheint. Sysmon und Registry-Monitoring sind daher unverzichtbare SOC-Tools.

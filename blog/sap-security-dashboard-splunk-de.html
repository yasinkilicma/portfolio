Überblick

SAP-Systeme speichern die sensibelsten Daten eines Unternehmens — Passwort-Hashes, RFC-Credentials, Rollen und Buchungsdaten. Trotzdem fehlt in vielen SOC-Teams ein dediziertes SAP-Monitoring-Dashboard. In diesem Projekt habe ich 128 realistische SAP Security Audit Log Events über 5 Angriffsszenarien simuliert und in einem einzigen Splunk-Dashboard sichtbar gemacht.

// SAP Security Monitoring Dashboard — Splunk Enterprise 10.2.3 — Home Lab

Lab-Architektur

Das Dashboard läuft auf dem gleichen Home-SOC-Lab wie meine anderen Analysen: Splunk Enterprise in einem Docker-Container, Log-Ingestion über den HTTP Event Collector (HEC) mit einem Python-Skript, das 128 SAP Audit-Events über 5 Szenarien generiert.

ARCHITEKTUR Datenfluss
# Log-Generierung Python Script (generate_sap_logs.py) └── 128 SAP Security Audit Events └── 5 Angriffsszenarien # Ingestion HTTPS POST → Splunk HEC (Port 8088) └── index=sap_audit └── sourcetype=sap:audit # Visualisierung Splunk Simple XML Dashboard └── 7 Zeilen × 2 Panels └── KPI / Chart / Table

SPL-Abfragen — Kernsätze

// Szenario 1 — Brute-Force

SPL AU2 ≥ 3 Fehlversuche in einer Stunde
index=sap_audit event_id=AU2 | eval event_hour=substr(timestamp,1,13) | stats count as Fehlversuche, values(terminal) as "Quell-IP" by user, event_hour | where Fehlversuche >= 3 | sort -Fehlversuche

// Szenario 2 — After-Hours Login

SPL Logins zwischen 22:00 und 06:00 Uhr
index=sap_audit (event_id=AU1 OR event_id=AU6) | eval hour=tonumber(substr(timestamp,12,2)) | eval is_after_hours=if(hour<6 OR hour>=22, "JA — KRITISCH", "Nein") | where is_after_hours="JA — KRITISCH" | table timestamp, user, event_id, transaction, terminal

// Szenario 3 — AUB Storm

SPL Benutzer mit ≥ 5 Berechtigungsfehlern
index=sap_audit event_id=AUB | stats count as "AUB-Events", values(transaction) as "Angegriffene TCodes", values(auth_object) as "Auth-Objekte" by user | where 'AUB-Events' >= 5

// Szenario 5 — Datenexfiltration

SPL EU1 — gelesene Zeilen pro Tabelle summieren
index=sap_audit event_id=EU1 | stats sum(rows_read) as "Gesamt-Datensätze", count as "Abfragen" by user, table | sort -"Gesamt-Datensätze"

Wichtige Erkenntnisse

⚠

Field-Mapping: SAP-Events im Splunk HEC-Format verwenden terminal für die Quell-IP (nicht src_ip), transaction für TCode (nicht transaction_code) und rows_read für gelesene Zeilen (nicht records_read). Immer zuerst mit | fields * prüfen.

→

EU1 + USR02/RFCDES: Wenn ein Benutzer die Tabellen USR02 (Passwort-Hashes) oder RFCDES (RFC-Credentials) ausliest, ist das ein sofortiger P1-Incident — unabhängig von der Anzahl der Zeilen.

✓

Dashboard als Vorlage: Das Simple XML liegt im Repository und kann direkt in jede Splunk-Instanz importiert werden — über Settings → User Interface → Views → Import oder via REST API.

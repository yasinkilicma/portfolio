Analyse 1: Cerber-Ransomware Detection

Suricata IDS Alert-Triage

🔴 Critical Finding

Suricata-Signatur "ETPRO TROJAN Ransomware/Cerber Onion Domain Lookup" identifiziert auf Host 192.168.250.20

Reconnaissance

  • Nessus-Scan: 37 Scans auf 192.168.250.20
  • Port-Scan: SMB (445) und NetBIOS (135)
  • 4 verschiedene IP-Ziele identifiziert

Detection

  • EventID: 1 (ProcessCreate)
  • Masquerading: osk.exe
  • .onion Domain Lookup (Tor/C2)
  • DNS-Exfiltration Indikator
# SPL Query: Suricata-Alerte gegen primäres Ziel
index=botsv1 sourcetype=suricata dest_ip=192.168.250.20
| stats count by alert.signature | sort -count

Analyse 2: SMB Lateral Movement

Network Forensics auf 151.568 SMB-Events

🔴 Ransomware Spread

192.168.250.100 → 192.168.250.20 mit 39.204 SMB-Events. Klare Indikation für Lateral Movement.

SMB-Protokoll

  • Port 445 (Windows File Sharing)
  • NTLM-Authentifizierung
  • Pass-the-Hash Angriffe
  • Admin-Shares (C$, ADMIN$)

Attack-Pfad

  • Credential Harvesting (Mimikatz)
  • Pass-the-Hash / Pass-the-Ticket
  • Remote Service Creation (PsExec)
  • File Encryption via SMB
# SPL Query: SMB-Verbindungen
index=botsv1 sourcetype=stream:smb
| stats count by src_ip, dest_ip | sort -count

Analyse 3: Sysmon Process Creation

Endpoint Forensics mit EventID 1

🔴 Masquerading Detected

osk.exe (On-Screen Keyboard) läuft aus C:\Users\...\AppData\Roaming\... — nicht aus System32!

Sysmon EventID 1

  • Image (Prozess-Pfad)
  • CommandLine (Argumente)
  • ParentImage (Eltern-Prozess)
  • Hashes (SHA1/MD5/SHA256)
  • IntegrityLevel

Verdächtige Prozesse

  • osk.exe (Masquerading)
  • php-cgi.exe (Webserver)
  • cmd.exe /c "dir 2>&1"
  • internat.exe (Persistence)
# XML-Parsing mit rex für Sysmon
index=botsv1 sourcetype=XmlWinEventLog...Operational
| rex field=_raw "<EventID>(?<EventID>\d+)</EventID>"
| search EventID=1
| rex field=_raw "<Data Name='Image'>(?<Image>[^<]+)</Data>"

Analyse 4: Registry Persistence

WinRegistry Run-Key Analyse

🔴 Persistence Mechanism

osk.exe in Registry Run und RunOnce Keys eingetragen. Auto-Start bei jedem Login.

Run-Keys

  • HKCU\...\Run\osk
  • HKCU\...\RunOnce\osk
  • HKCU\...\Run\internat.exe
  • Command Processor\AutoRun

Technik

  • Registry-Run-Keys (Auto-Start)
  • Masquerading (legitime Namen)
  • CMD-Hijacking (AutoRun)
  • User-Profile Persistence
# SPL Query: Registry-Run-Keys
index=botsv1 sourcetype=WinRegistry
| search key_path="*Run*"
| stats count by key_path, data

Analyse 5: Fortigate Firewall Traffic

Perimeter Analysis auf 55.279 Events

🔴 C2-Kommunikation

192.168.250.20 → 8.8.8.8 (Google DNS) mit 1.194 ACCEPT Events. DNS-basierte C2-Kommunikation ungehindert.

Gefundene Patterns

  • accept: C2-DNS (1.194)
  • close: Externe IPs (1.438)
  • deny: Broadcast (1.498)
  • start: Scanner-DNS (4.080)

Firewall-Actions

  • accept = Verbindung erlaubt
  • deny = Paket blockiert
  • close = Session beendet
  • start = Session gestartet
# SPL Query: Fortigate Traffic
index=botsv1 sourcetype=fgt_traffic
| stats count by srcip, dstip, action | sort -count

Security Assessment

Wo die Firewall versagte

⚠️ Nur 20% Effektivität

Fortigate erfüllte nur Basis-Funktion (Broadcast-Denies). Kritische Bedrohungen passierten ungehindert.

🔴 Gaps (Versagen)

  • C2-DNS: ACCEPT (kein DNS-Filter)
  • SMB: Unsichtbar (kein East-West)
  • Masquerading: Blind (kein EDR)
  • IPS: Keine SMB-Signatures

✅ Erfolg (Baseline)

  • Broadcast-Denies (Standard)
  • Session-Logging (55.279 Events)
  • Port-basiertes Blocking
# Defense-in-Depth: 5 Schichten nötig
1. DNS Security (FortiGuard DNS)
2. Internal Segmentation (VLAN)
3. IPS Signatures (SMB/Exploits)
4. Endpoint Detection (Sysmon/EDR)
5. Network Detection (Zeek/Suricata)

Key Takeaways

  • Multi-Source-Correlation: Suricata + Sysmon + Registry + Firewall = vollständiges Bild
  • Firewall ≠ 100% Sicherheit: Perimeter allein reicht nicht für Lateral-Movement
  • DNS-Monitoring essenziell: C2 nutzt DNS als Tunnel — Port 53 ist nicht harmlos
  • Endpoint Logs kritisch: Ohne Sysmon bleibt Masquerading unsichtbar
  • SPL ist mächtig: XML-Parsing mit rex, Aggregation, Filterung

Warum ich für SOC geeignet bin

Von Remote-Hands zu SOC Analyst

🎯

Meine aktuelle Arbeit ist der Hardware-Teil des SOC-Workflows: Monitoring-Alarm → prüfen → handeln → dokumentieren. Genau derselbe Zyklus, nur mit physischen Geräten statt Security-Events.

Meine aktuelle Erfahrung

  • Hardware IMAC — Disk-Einbau (12/48 Disks in Cluster), Komponentenwechsel, Change-Request-Management
  • Network Device Tasks — Cisco Nexus Switch-Reboots, physische Interventions
  • Monitoring-Alarm Incident — Netzwerkgerät reagiert nicht auf Monitoring → Alarm → Vor-Ort-Check + Reboot + Dokumentation
  • ITSM-Disziplin — ServiceNow-Tickets, Priorisierung, Eskalation, SLA-Tracking
  • Schichtübergabe — Lückenlose Übergabe für 24/7-Betrieb

→ SOC-Parallelen

  • Incident-Response-Zyklus → Alarm → Triaging → Eskalation → Dokumentation (birebir aynı)
  • Ticket-Disziplin → ServiceNow-Skills direkt übertragbar
  • Kritische Systeme unter Druck → Nightly-Maintenance, SLA-Druck, präzise Abläufe
  • Compliance/Chain of Custody → Datenträger dürfen DC nicht verlassen (Drop-Box + SN-Tracking) → vergleichbar mit Beweissicherung im IR
  • Prozessdisziplin → SAP-Change-Approval vor jedem Reboot/Shutdown
  • 24/7-Mindset → Schichtübergabe, Kontinuität, Verantwortung
Goldener Satz für das Gespräch:

"Monitoring-Alarm → prüfen → handeln → dokumentieren — das ist im Kern genau der SOC-Workflow, nur mit Hardware statt Security-Events. Und ITSM-Disziplin, 24/7-Bereitschaft und Compliance-Regeln kenne ich bereits aus der Praxis."
